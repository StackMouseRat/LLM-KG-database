## 案例文档特征

### 内容结构
每个案例通常包含：
1. **异常情况描述**
   - 时间、地点、设备信息
   - 故障现象
   - 监控数据、报警信号

2. **分析处理过程**
   - 初步判断
   - 检查步骤
   - 测量数据
   - 原因分析

3. **处理结果**
   - 解决方案
   - 处理效果
   - 验证测试

4. **结论与预防措施**
   - 故障根本原因
   - 预防建议
   - 维护要点

### 数据类型
- 设备型号、参数
- 电气测量数据（电压、电流、电阻等）
- 油色谱分析数据
- 温度、压力等物理量
- 图表、照片说明
- 时间序列数据

## 使用建议

### FastGPT 导入参数
```json
{
  "分块方式": "智能分块",
  "chunk_size": 500-800,
  "chunk_overlap": 100,
  "搜索方式": "语义检索",
  "最低相关度": 0.70,
  "引用上限": 1800
}
```

### 分批导入策略
1. **第一批**（1周内）：优先级1的5个文件，验证效果
2. **第二批**（2周内）：优先级2的5个文件，扩展覆盖
3. **第三批**（3周内）：优先级3的7个文件，完善细节

### 检索场景
- ✅ 相似故障案例查询："变压器油色谱异常怎么处理？"
- ✅ 处理经验参考："断路器拒合的常见原因有哪些？"
- ✅ 预防措施建议："如何预防互感器爆炸？"
- ✅ 异常现象解释："潜油泵电流不平衡是什么原因？"

## 与知识图谱的配合

### 知识图谱（Nebula）
- **用途**：结构化知识推理
- **优势**：精确查询、多跳推理、关系明确
- **数据源**：`txt/` 目录下的层级梳理文档（73个）

### RAG 案例库（FastGPT）
- **用途**：案例检索、经验参考
- **优势**：模糊匹配、隐性知识、处理细节
- **数据源**：本目录下的案例文档（17个）

### 混合架构
```
用户提问
    ↓
需求解析（LLM）
    ↓
    ├─→ KG 查询：故障类型→原因→措施（结构化）
    │
    ├─→ RAG 检索：相似历史案例（非结构化）
    │
    └─→ 模板匹配：预案框架
    ↓
内容融合（LLM）
    ↓
预案生成
```
