# 设备故障图谱通用工作流程

## 1. 文档定位

本流程用于后续维护不同设备类型的故障图谱数据，不限定为输电线路，也适用于变压器、断路器、电缆、绝缘子等其他设备。

这份流程强调两件事：

- 图谱生产方式应通用
- 节点数量、层级数量、关系数量不应写死

后续每做一种新设备，都应先形成该设备自己的“层级汇总稿”，再生成对应 Excel 交付包。

## 2. 可复用的固定项与可配置项

### 2.1 固定项

以下内容建议作为通用规则长期保留：

- 先整理文字版层级稿，再生成图谱数据
- 节点命名要统一，避免同义词和临时名称
- 关系口径要提前定稿，避免边生成边改语义
- 交付物仍以 NRD Studio 所需的 Excel 文件集为准
- 生成后必须做结构校验和字段校验

### 2.2 可配置项

以下内容不应写死，必须按具体设备单独确定：

- 设备名称
- 根节点数量
- 一级分类数量
- 二级分类数量
- 是否继续细分三级、四级结构
- 详情节点类型
- 每类关系是否启用
- 分组方式
- 标签和区块是否启用
- 最终节点数、关系数、圈子数

## 3. 推荐建模框架

不同设备可以共用同一套建模方法，但不要求节点总数一致。

推荐先按“设备实体 -> 故障分类 -> 详情说明”的思路建模，再根据业务复杂度裁剪。

默认推荐框架如下：

1. 设备实体
2. 一级故障大类
3. 二级故障小类
4. 详情节点

详情节点类型可按项目决定，常见可选项包括：

- 故障原因
- 故障现象
- 应对措施
- 故障后果
- 安全风险
- 应急资源

说明：

- 有些设备只需要 `故障原因 + 故障现象 + 应对措施`
- 有些设备需要完整保留 6 类详情节点
- 如果某类设备还需要“检测方法”“典型案例”“影响部位”等节点，也可以继续扩展

## 4. 标准工作流程

### 步骤 1：整理原始资料

原始资料通常来自：

- `docs/` 中的设备资料、故障分析文档
- `txt/` 中的摘录稿、整理稿、汇总稿
- `img/` 中的结构图或 OCR 辅助图片

先完成以下工作：

- 统一设备名称和故障术语
- 去重合并重复表述
- 区分“分类节点”和“描述内容”
- 明确哪些内容应成为节点，哪些内容只写入描述字段

### 步骤 2：先写设备专属层级汇总稿

每种设备都应单独维护一份汇总稿，建议命名为：

- `txt/<设备类型>/<设备类型>故障层级汇总稿.md`

汇总稿至少应包含以下内容：

- 建模范围
- 层级结构
- 分类清单
- 关系口径
- 命名规则
- 各类详情节点的描述内容

### 步骤 3：确定本设备的配置表

在生成图谱前，先明确本轮项目配置：

- 根节点是否只有 1 个
- 一级分类有多少类
- 二级小类有多少类
- 每个二级小类是否都需要完整详情节点
- 是否允许某些小类缺少某类详情节点
- 是否需要分组
- 是否需要标签
- 是否需要区块

建议把这一步当作“设备级参数表”，不要直接沿用其他设备的数量。

### 步骤 4：执行命名归一

命名应遵循以下原则：

- 一级和二级名称使用业务上稳定的正式名称
- 同一概念只保留一个名称
- 不使用 `原因1`、`措施2`、`现象A` 这类临时名称
- 如果启用详情节点，建议统一采用 `<故障小类>-<层级名>` 的唯一命名方式

### 步骤 5：整理详情节点内容

对每个故障小类，逐项判断是否需要以下节点：

- 故障原因
- 故障现象
- 应对措施
- 故障后果
- 安全风险
- 应急资源

处理规则如下：

- 需要该节点时，写入 1 个节点并把详细内容合并到 `desc`
- 不需要该节点时，可以不生成，不强制补齐
- 原始资料没有明确信息时，只做最小归纳

### 步骤 6：映射为图谱结构

在结构化落图时，应先画出本设备的关系清单，而不是直接照搬其他设备的数量。

推荐做法：

1. 先列出本设备全部节点类型
2. 再列出每种关系的起点、终点和语义
3. 最后再计算理论节点数和关系数

如果一个节点允许属于多个分组，则：

- `圈子成员_group_members.xlsx` 的行数可能大于“非根节点总数”

因此，圈子成员数也不应写死。

### 步骤 7：生成 Excel 交付包

交付包仍建议保持统一文件形态，便于后续维护和比对。

推荐输出文件如下：

- `节点_nodes.xlsx`
- `关系_links.xlsx`
- `圈子_groups.xlsx`
- `圈子成员_group_members.xlsx`
- `区块_blocks.xlsx`
- `标签_tags.xlsx`
- `标签成员_tag_members.xlsx`

说明：

- 其中 `区块`、`标签`、`标签成员` 可以为空表
- 这 7 个文件是交付形态，不代表每个项目都必须填满所有业务内容

### 步骤 8：生成后校验

校验时不要只看“文件生成成功”，要检查以下内容：

- 文件是否齐全
- 表头是否符合要求
- 节点主键和关系引用是否一致
- `from/to/member_id` 是否都能在节点表中找到
- 分组名称是否和分组成员表一致
- 关系类型是否只使用本设备定义好的口径
- 节点总数、关系总数是否与本轮配置表一致

### 步骤 9：归档

归档时建议同时保留三类文件：

1. 设备专属汇总稿
2. 最新生成结果
3. 最终确认版成品

推荐目录形式：

- `txt/<设备类型>/<设备类型>故障层级汇总稿.md`
- `xls/<设备类型>/`
- `xls/<设备类型>/<设备类型>_图谱数据/`
- `xls/<设备类型>/成品/<设备类型>_图谱数据/`

## 5. 通用校验方式

后续不同设备的校验建议统一按“公式 + 配置表”执行。

### 5.1 节点数

节点总数应按实际设计计算：

`节点总数 = 根节点数 + 分类节点数 + 详情节点数`

其中：

- 分类节点数 = 各级分类节点之和
- 详情节点数 = 所有故障小类实际启用的详情节点之和

### 5.2 关系数

关系总数应按本轮启用的关系规则计算：

`关系总数 = 各类关系实际生成条数之和`

不要预设所有设备都和某个旧项目相同。

### 5.3 圈子成员数

圈子成员数的计算规则取决于分组策略：

- 如果每个节点只属于 1 个圈子，则成员数通常接近“被分组节点总数”
- 如果允许一个节点属于多个圈子，则成员数会大于被分组节点总数
- 如果某些节点不进圈子，则成员数会小于节点总数

## 6. Excel 交付结构建议

虽然业务数量不固定，但 Excel 结构最好保持稳定。

当前建议保留以下主表字段：

`节点_nodes.xlsx`

- `id`
- `name`
- `degree`
- `desc`
- `fontFamily`
- `fontSize`
- `image`
- `lineWidth`
- `r`
- `stroke`
- `textBorderWidth`
- `weight`

`关系_links.xlsx`

- `id`
- `from`
- `fromNodeName`
- `lineWidth`
- `relation`
- `stroke`
- `textBorderWidth`
- `to`
- `toNodeName`

`圈子_groups.xlsx`

- `id`
- `name`
- `desc`
- `member_count`
- `order`
- `show`
- `type`
- `itemStyle.fill`
- `itemStyle.markColor`
- `itemStyle.stroke`

`圈子成员_group_members.xlsx`

- `id`
- `name`
- `member_id`
- `member_name`

## 7. 当前仓库中的设备示例

当前仓库现有脚本和产物以“输电线路”为示例。

这个示例可以作为参考，但不能把它的数量当成通用规则：

- 当前输电线路成品节点数是特定项目结果，不是固定模板
- 当前输电线路关系数也是特定关系设计的结果，不是所有设备都应相同
- 后续新设备应沿用方法，不应照搬数量

## 8. 一句话版流程

后续统一按这条主线执行：

`整理原始资料 -> 编写设备专属层级汇总稿 -> 确定本设备配置表 -> 统一命名 -> 生成结构化节点与关系 -> 输出 Excel -> 按本轮配置表校验 -> 归档`
