# 电力设备图谱建表与导入流程指南

## 1. 目标

本文定义从原始资料到 Nebula 图数据库的通用建表与导入流程，适用于电力电缆、断路器、互感器、变压器、输电线路，以及后续新增设备。

本流程采用当前正式库的新结构：

- 顶点保留兼容 tag：`entity`
- 顶点同时写入 typed tags
- 边使用语义化 edge types
- 主查询不再依赖旧的统一 `rel(relation)` 边

旧 `rel` 可在兼容期保留，但新增或重建设备应直接按本文的新结构落库。

## 2. 通用目录约定

以 `<设备名>` 作为目录名，例如 `电力电缆`、`高压断路器`、`变压器`。

### 2.1 原始资料

原始资料可放在：

- `docs/<设备名>.docx`
- `docs/<设备名>*.docx`
- `txt/<设备名>/` 中已有的人工整理材料

### 2.2 纯文本与层级稿

统一输出到：

- `txt/<设备名>/`

建议文件清单：

- `<设备名>.txt`
- `<设备名>故障层级梳理_第一层级.md`
- `<设备名>故障层级梳理_第二层级.md`
- `<设备名>故障层级梳理_第三层级_故障原因.md`
- `<设备名>故障层级梳理_第四层级_故障现象.md`
- `<设备名>故障层级梳理_第五层级_应对措施.md`
- `<设备名>故障层级梳理_第六层级_故障后果.md`
- `<设备名>故障层级梳理_第七层级_安全风险.md`
- `<设备名>故障层级梳理_第八层级_应急资源.md`
- `<设备名>故障层级梳理_第一至第八层级汇总稿.md`

## 3. 通用建模口径

### 3.1 层级定义

默认采用 8 层结构：

1. 根节点：设备本体
2. 一级故障大类
3. 二级故障小类
4. 故障原因
5. 故障现象
6. 应对措施
7. 故障后果
8. 安全风险 / 应急资源

在 Nebula 中，详情节点不按文档编号强绑定，而按 typed tag 表达：

- `fault_cause`
- `fault_symptom`
- `response_measure`
- `fault_consequence`
- `safety_risk`
- `emergency_resource`

### 3.2 二级故障整理原则

二级故障必须尽量“不重不漏”：

- 二级节点只保留故障结果、故障对象或稳定故障形态
- 不把原因、诱因、检测过程、处置过程作为二级故障
- 同类原因造成的同一故障结果应合并
- 制造缺陷、安装缺陷、设计不良等通常进入“故障原因”，不单独作为二级故障
- 诊断定位异常通常进入“应对措施”或“应急资源”，不作为设备故障二级项

### 3.3 详情节点命名

每个二级故障小类下，建议保留 6 类详情节点：

- `<二级故障小类>-故障原因`
- `<二级故障小类>-故障现象`
- `<二级故障小类>-应对措施`
- `<二级故障小类>-故障后果`
- `<二级故障小类>-安全风险`
- `<二级故障小类>-应急资源`

每类详情节点只保留一个节点，详细内容合并写入 `node_desc`。

## 4. Nebula 目标结构

### 4.1 顶点 tag

每个节点写入 `entity`，同时按类型写入对应 typed tag：

- `root_node`
- `fault_l1`
- `fault_l2`
- `fault_cause`
- `fault_symptom`
- `response_measure`
- `fault_consequence`
- `safety_risk`
- `emergency_resource`

字段统一为：

```ngql
name string,
node_desc string,
lvl int,
source_id int,
degree int,
weight int,
stroke string
```

### 4.2 语义边

主查询使用语义化 edge types：

- `has_fault_category`
- `contains`
- `caused_by`
- `has_symptom`
- `handled_by`
- `results_in`
- `has_risk`
- `needs_resource`

对应关系口径：

```text
root_node -[:has_fault_category]-> fault_l1
fault_l1 -[:contains]-> fault_l2
fault_l2 -[:caused_by]-> fault_cause
fault_cause -[:has_symptom]-> fault_symptom
fault_l2 -[:handled_by]-> response_measure
fault_cause -[:results_in]-> fault_consequence
response_measure -[:has_risk]-> safety_risk
response_measure -[:needs_resource]-> emergency_resource
```

## 5. xls 生成规范

### 5.1 输出目录

工作目录：

- `xls/<设备名>/<设备名>_第一至第八层级图谱数据/`

成品目录：

- `xls/成品/<设备名>/<设备名>_第一至第八层级图谱数据/`

### 5.2 输出文件

每个设备应生成：

- `节点_nodes.xlsx`
- `关系_links.xlsx`
- `圈子_groups.xlsx`
- `圈子成员_group_members.xlsx`
- `区块_blocks.xlsx`
- `标签_tags.xlsx`
- `标签成员_tag_members.xlsx`
- `graph_manifest.json`

其中：

- Excel 文件用于交付和人工检查
- `graph_manifest.json` 用于 Nebula 导入

### 5.3 manifest 必备字段

`graph_manifest.json` 至少应包含：

```json
{
  "space": "llmkg_xxx",
  "root": "设备名",
  "nodes": [
    {
      "vid": "n_0",
      "name": "设备名",
      "node_desc": "描述",
      "lvl": 0,
      "source_id": 0,
      "degree": 1,
      "weight": 10,
      "stroke": "#9A3412",
      "typed_tag": "root_node"
    }
  ],
  "links": [
    {
      "src": "n_0",
      "dst": "n_1",
      "edge": "has_fault_category"
    }
  ]
}
```

如果 manifest 中仍使用中文关系名，应在导入前映射为语义 edge type。

## 6. Nebula 导入规范

### 6.1 建 schema

导入脚本必须确保以下 tag 存在：

```ngql
CREATE TAG IF NOT EXISTS entity(...);
CREATE TAG IF NOT EXISTS root_node(...);
CREATE TAG IF NOT EXISTS fault_l1(...);
CREATE TAG IF NOT EXISTS fault_l2(...);
CREATE TAG IF NOT EXISTS fault_cause(...);
CREATE TAG IF NOT EXISTS fault_symptom(...);
CREATE TAG IF NOT EXISTS response_measure(...);
CREATE TAG IF NOT EXISTS fault_consequence(...);
CREATE TAG IF NOT EXISTS safety_risk(...);
CREATE TAG IF NOT EXISTS emergency_resource(...);
```

必须确保以下 edge 存在：

```ngql
CREATE EDGE IF NOT EXISTS has_fault_category();
CREATE EDGE IF NOT EXISTS contains();
CREATE EDGE IF NOT EXISTS caused_by();
CREATE EDGE IF NOT EXISTS has_symptom();
CREATE EDGE IF NOT EXISTS handled_by();
CREATE EDGE IF NOT EXISTS results_in();
CREATE EDGE IF NOT EXISTS has_risk();
CREATE EDGE IF NOT EXISTS needs_resource();
```

### 6.2 写入规则

每个节点必须写入两类 tag：

- `entity`
- 对应 typed tag

每条边只写入语义 edge type。

新增设备不再写 `rel`。旧设备兼容期可以保留 `rel`，但新查询不应依赖它。

### 6.3 稳定性要求

当前网关对 Nebula 错误会包装为 `502`，因此导入脚本必须：

- 对 HTTP 5xx 做重试
- 顶点插入使用小批量或单条写入
- 大段中文文本必须正确转义双引号、反斜杠和换行
- 避免依赖 `GROUP BY`
- 避免对无索引边属性直接 `WHERE e.relation == ...`

## 7. 通用验证查询

### 7.1 顶点计数

```ngql
MATCH (v:entity) RETURN count(v) AS c;
MATCH (v:root_node) RETURN count(v) AS c;
MATCH (v:fault_l1) RETURN count(v) AS c;
MATCH (v:fault_l2) RETURN count(v) AS c;
MATCH (v:fault_cause) RETURN count(v) AS c;
MATCH (v:fault_symptom) RETURN count(v) AS c;
MATCH (v:response_measure) RETURN count(v) AS c;
MATCH (v:fault_consequence) RETURN count(v) AS c;
MATCH (v:safety_risk) RETURN count(v) AS c;
MATCH (v:emergency_resource) RETURN count(v) AS c;
```

### 7.2 边计数

```ngql
MATCH ()-[e:has_fault_category]->() RETURN count(e) AS c;
MATCH ()-[e:contains]->() RETURN count(e) AS c;
MATCH ()-[e:caused_by]->() RETURN count(e) AS c;
MATCH ()-[e:has_symptom]->() RETURN count(e) AS c;
MATCH ()-[e:handled_by]->() RETURN count(e) AS c;
MATCH ()-[e:results_in]->() RETURN count(e) AS c;
MATCH ()-[e:has_risk]->() RETURN count(e) AS c;
MATCH ()-[e:needs_resource]->() RETURN count(e) AS c;
```

### 7.3 一级故障目录

```ngql
MATCH (r:root_node)-[:has_fault_category]->(l1:fault_l1)
RETURN l1.fault_l1.source_id AS source_id,
       l1.fault_l1.name AS name,
       l1.fault_l1.node_desc AS node_desc
ORDER BY source_id;
```

### 7.4 一级到二级目录

```ngql
MATCH (l1:fault_l1)-[:contains]->(l2:fault_l2)
RETURN l1.fault_l1.source_id AS l1_source_id,
       l1.fault_l1.name AS l1_name,
       l2.fault_l2.source_id AS l2_source_id,
       l2.fault_l2.name AS l2_name
ORDER BY l1_source_id, l2_source_id;
```

### 7.5 某二级故障完整详情

```ngql
MATCH (f:fault_l2)
WHERE f.fault_l2.name CONTAINS "关键词"
OPTIONAL MATCH (f)-[:caused_by]->(c:fault_cause)
OPTIONAL MATCH (c)-[:has_symptom]->(s:fault_symptom)
OPTIONAL MATCH (f)-[:handled_by]->(m:response_measure)
OPTIONAL MATCH (c)-[:results_in]->(co:fault_consequence)
OPTIONAL MATCH (m)-[:has_risk]->(r:safety_risk)
OPTIONAL MATCH (m)-[:needs_resource]->(er:emergency_resource)
RETURN f.fault_l2.name AS fault,
       c.fault_cause.node_desc AS cause,
       s.fault_symptom.node_desc AS symptom,
       m.response_measure.node_desc AS measure,
       co.fault_consequence.node_desc AS consequence,
       r.safety_risk.node_desc AS risk,
       er.emergency_resource.node_desc AS resource;
```

## 8. 旧表升级到新模式

旧表如果已经存在 `entity + rel(relation)`，升级步骤为：

1. 先补齐 typed tags
2. 创建 8 类语义 edge type
3. 扫描旧 `rel` 边
4. 将中文关系映射为语义 edge type
5. 写入新语义边
6. 保留旧 `rel` 兼容
7. 查询端切换到语义 edge type
8. 稳定后再考虑删除旧 `rel`

中文关系映射：

```text
发生   -> has_fault_category
包含   -> contains
起因于 -> caused_by
表现   -> has_symptom
表现为 -> has_symptom
处置   -> handled_by
导致   -> results_in
存在   -> has_risk
需要   -> needs_resource
```

当前升级脚本：

- `scripts/upgrade_semantic_edges_current_schema.py`

## 10. 注意事项

- 主查询必须优先使用 typed tags 和语义 edge types。
- 新增设备不应再写旧 `rel`。
- 旧正式表可以在兼容期保留 `rel`。
- 网关返回 `502` 不一定代表数据库不可用，可能是 Nebula 语法错误、索引错误、console 连接失败或超时。
- 大段中文写入不建议大批量插入，推荐单条或小批量。
- 生成脚本和导入脚本应按设备单独维护，但 schema 和验证口径必须统一。
