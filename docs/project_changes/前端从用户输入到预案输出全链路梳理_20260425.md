# 前端从用户输入到预案输出全链路梳理

## 1. 先说结论

基于当前仓库里已经落地的前端、部署代理、流水线脚本和已保存的 FastGPT 原始响应，当前“预案生成”主链路的实际工作方式是：

1. 用户在前端 `预案生成` 页输入自然语言故障问题。
2. 前端通过 `/api/pipeline/run` 发起流式请求。
3. 当前实际部署代理 `docker/frontend-proxy/server.py` 不直接调用 FastGPT，而是编排 `scripts/run_parallel_generation_pipeline.py`、案例检索插件和其他后端能力。
4. Python 流水线依次调用 3 类 FastGPT 插件：
   - `基本信息获取`
   - `模板切片`
   - `并行生成`
5. `并行生成` 不是整篇一次生成，而是按章节并发调用，每章各调用一次插件。
6. 前端通过 SSE 持续接收阶段事件和章节增量文本，边生成边渲染。
7. 全部章节完成后，前端把结果缓存到浏览器本地，并支持复制、下载、进入图谱溯源/质量评估等后续页面。

要特别注意两点：

- 当前仓库里同时存在两套代理：
  - `frontend/server/proxy.mjs`：偏本地开发/简化代理
  - `docker/frontend-proxy/server.py`：当前实际部署代理
- 前端里的 `案例搜索` 和 `多故障检索` 在当前实际部署链路中都已经接通：
  - `案例搜索` 由 `server.py` 并发调用知识库案例检索插件
  - `多故障检索` 由 `server.py` 把前端开关转换成脚本参数 `--multi-fault`

## 2. 前端侧从输入到结果展示经过了什么

### 2.1 用户输入阶段

入口页是 `frontend/src/App.tsx` 中的 `/plan` 页面。

用户在这里可以做的事：

- 输入故障问题 `question`
- 随机填充一个预置问题
- 勾选 `案例搜索`
- 勾选 `多故障检索`

当前这 3 项都会影响后端执行：

- `question`
- `enableCaseSearch`
- `enableMultiFaultSearch`

### 2.2 前端发起请求

前端通过 `frontend/src/services/planApi.ts` 的 `runPipelineStream()` 调用：

- 请求地址：`/api/pipeline/run`
- 请求方式：`POST`
- 请求头：`Accept: text/event-stream`
- 请求体：

```json
{
  "question": "用户输入的问题",
  "enableCaseSearch": false,
  "enableMultiFaultSearch": false,
  "stream": true
}
```

这里前端会把两个开关也一并传给部署代理，部署代理会真实读取并进入对应执行分支。

### 2.3 代理层处理

当前实际部署代理是 `docker/frontend-proxy/server.py`。

它收到请求后会读取：

- `question`
- `enableCaseSearch`
- `enableMultiFaultSearch`

并按这两个开关决定后端执行方式：

1. 始终调用主流水线脚本 `scripts/run_parallel_generation_pipeline.py`
2. 如果开启 `多故障检索`，则为脚本追加：

```bash
--multi-fault
```

3. 如果开启 `案例搜索`，则并发调用知识库案例检索插件 `run_case_search(...)`

因此这两个 UI 开关在当前实际部署链路里都是真正参与执行分支的。

### 2.4 SSE 回传给前端

代理会把 Python 脚本输出的事件重新包装成 SSE 返给前端。

当前前端已经识别的事件包括：

- `basic_info_started`
- `basic_info_done`
- `template_split_started`
- `template_split_done`
- `parallel_generating_started`
- `chapter_started`
- `chapter_chunk`
- `chapter_done`
- `case_search_started`
- `case_search_done`
- `case_search_error`
- `pipeline_done`
- `pipeline_error`

其中在当前实际部署链路里，`case_search_started / done / error` 也会真实出现。

### 2.5 页面如何展示

前端收到事件后的行为是：

- `basic_info_done` 后更新摘要信息：
  - `userQuestion`
  - `faultScene`
  - `graphMaterial`
- `template_split_done` 后创建章节卡片骨架
- `chapter_chunk` 时把某一章的文本增量追加到当前卡片
- `chapter_done` 时标记该章完成
- `pipeline_done` 时组装最终 `PipelineRunResponse`

前端完成后还会：

- 把最近一次成功结果写入 `localStorage`
- 支持复制完整 Markdown
- 支持下载为 Markdown
- 为 `/trace`、`/quality`、`/template` 等后续页面提供输入上下文

## 3. 后端主流水线实际经过的环节

当前真实主链路由“部署代理 + Python 流水线”共同控制，严格顺序如下：

### 3.1 阶段一：基本信息获取

调用 FastGPT 插件 `基本信息获取`。

流水线传入：

```json
{
  "用户问题": "用户原始问题"
}
```

脚本从该插件返回结果里抽取并标准化成：

```json
{
  "用户问题": "...",
  "故障与场景提取结果": "...",
  "图谱检索方案素材": "...",
  "模板文本": ""
}
```

这一步结束后，用户自然语言问题已经不再只是原始文本，而是被转成了“结构化场景 + 图谱素材”的中间表示。

### 3.2 阶段二：模板切片

调用 FastGPT 插件 `模板切片`。

这一阶段不需要传变量，直接取当前模板定义。

产出的是：

```json
{
  "template_id": "tpl_default_emergency",
  "template_name": "默认应急预案模板",
  "current_version": "v1",
  "chapter_count": 6,
  "chapters": [
    {
      "chapter_no": "1",
      "title": "事件特征",
      "section_count": 6,
      "text": "当前章节模板文本"
    }
  ]
}
```

这一步的作用不是写预案，而是把整份模板拆成多个章节任务。

### 3.3 阶段三：并行章节生成

脚本会对 `chapters[]` 中每一章分别调用一次 `并行生成` 插件。

每章传入变量固定为：

```json
{
  "用户问题": "...",
  "故障与场景提取结果": "...",
  "图谱检索方案素材": "...",
  "模板": "该章节模板文本"
}
```

然后：

- 使用线程池并行发起多章调用
- 每章流式接收 `answer` 增量
- 把文本按 `chapter_chunk` 持续回推给前端
- 每章完成后产出 `output_text`

最后再按章节号排序，写入：

- `pipeline_result.json`
- `pipeline_result.md`

### 3.4 并行案例检索

如果前端开启 `案例搜索`，部署代理 `docker/frontend-proxy/server.py` 会并发执行知识库案例检索。

它的执行方式有两种：

1. 若代理能在请求刚进入时就根据问题推断设备数据集，则直接与主流水线并发启动
2. 若一开始无法稳定判断，则在收到 `basic_info_done` 之后，结合 `faultScene` 和 `graphMaterial` 再启动案例检索

完成后通过 SSE 回传：

- `case_search_started`
- `case_search_done`
- `case_search_error`

### 3.5 多故障分支切换

如果前端开启 `多故障检索`，部署代理会把前端参数 `enableMultiFaultSearch` 转成脚本参数：

```bash
--multi-fault
```

这样主流水线第一阶段就不再走单故障插件，而是切换到多故障链路。

## 4. 每个 FastGPT 插件到底是怎么工作的

这里按“当前前端主链路会接触到的插件”和“脚本已支持但前端未真正接上的插件”两类来讲。

### 4.1 插件一：`基本信息获取`

当前状态：

- 是当前前端主链路中的真实第一段
- 已有真实原始响应样本
- 发布版信息见 `docs/对照试验题库/现有工作流生成链路梳理.md`

#### 4.1.1 它的职责

把用户自然语言问题转成后续生成可直接使用的中间素材：

- 识别该问题应该落到哪个设备图谱空间
- 从该设备空间里匹配最合理的二级故障
- 抽取故障对象、事件场景、关键处置要求、特殊约束等场景字段
- 从 Nebula 图谱中查询该故障相关的原因、现象、措施、后果、风险、资源
- 最终输出统一的图谱检索素材文本

#### 4.1.2 它的内部节点工作方式

从保存的真实响应看，这个插件当前内部是“规则节点 + HTTP 查询 + 代码清洗 + 内容抽取”的组合，不是单一 LLM。

内部主链如下：

1. `pluginInput`
2. `ifElseNode`
   - 节点名：`设备表初始化`
   - 作用：判断是否已有设备空间上下文
3. `httpRequest468`
   - 节点名：`获取全体设备表`
   - 作用：对 Nebula 执行 `SHOW SPACES;`
4. `code`
   - 节点名：`清洗设备表数据`
   - 作用：把原始 `SHOW SPACES` 输出整理成逗号分隔的设备空间列表
5. `contentExtract`
   - 节点名：`设备识别`
   - 作用：从设备空间列表中选出最匹配当前问题的设备表，例如：
     - `llmkg_breaker`
     - `llmkg_transmission_line`
     - `llmkg_cable`
6. `httpRequest468`
   - 节点名：`基础数据获取`
   - 作用：查询该设备空间下的一级/二级故障目录
7. `code`
   - 节点名：`基础数据清洗`
   - 作用：把故障目录整理成给抽取模型使用的目录文本
8. `contentExtract`
   - 节点名：`故障类型分析`
   - 作用：从目录中匹配最合适的二级故障，并输出结构化 JSON
9. `httpRequest468`
   - 节点名：`下游节点获取`
   - 作用：按命中的二级故障去图谱里查询：
     - `fault_cause`
     - `fault_symptom`
     - `response_measure`
     - `fault_consequence`
     - `safety_risk`
     - `emergency_resource`
10. `code`
    - 节点名：`下游节点数据清洗`
    - 作用：把图查询结果拼成统一文本
11. `pluginOutput`

#### 4.1.3 它用到 LLM 的位置

只有两个 `contentExtract` 环节使用模型：

- `设备识别`
- `故障类型分析`

也就是说，这个插件不是“直接让大模型写一段解释”，而是先把问题映射成：

- 设备空间
- 二级故障
- 场景 JSON

再去做图谱查询。

#### 4.1.4 它的输出长什么样

插件原始输出字段主要是：

- `用户问题`
- `故障类型分析`
- `图谱检索`
- `知识库名`

在 Python 流水线里又被统一改名为：

- `用户问题`
- `故障与场景提取结果`
- `图谱检索方案素材`
- `模板文本`

#### 4.1.5 它对后续生成的意义

后面的 `并行生成` 实际上并不是直接读用户问题，而是高度依赖这一阶段产出的：

- 场景结构化结果
- 图谱检索素材

所以这一步是整个系统里“自然语言转可执行生成条件”的关键桥梁。

---

### 4.2 插件二：`模板切片`

当前状态：

- 是当前前端主链路中的真实第二段
- 已有完整原始响应样本

#### 4.2.1 它的职责

从模板图空间读取“当前启用模板”的全部章节定义，并把它转成章节级文本任务。

#### 4.2.2 它的内部节点工作方式

这个插件当前不是生成类插件，内部没有 `chatNode`。

真实内部主链如下：

1. `pluginInput`
2. `httpRequest468`
   - 节点名：`HTTP 请求`
   - 作用：查询 Nebula 图空间 `llmkg_templates`
   - 读取内容包括：
     - `template_id`
     - `template_name`
     - `current_version`
     - `section_no`
     - `title`
     - `level`
     - `source_type`
     - `kg_field`
     - `fixed_text`
     - `gen_instruction`
3. `code`
   - 节点名：`代码运行`
   - 作用：
     - 过滤启用章节
     - 按 `order_no` 排序
     - 组装章节树
     - 拼出每章的 `text`
4. `pluginOutput`

#### 4.2.3 它为什么重要

它决定了并行生成插件到底收到什么样的“章节模板文本”。

它会把每章拼成这种形式：

```text
## 第4章 处置措施
内容来源：kg+gen
图谱字段：response_measure
生成要求：结合风险与场景扩写具体措施
```

后面再继续拼子节和小节。

所以真正给模型的并不是一个抽象 JSON 模板，而是一段已经展开好的“章节写作说明文本”。

#### 4.2.4 它的本质

这个插件本质上是“模板读取器 + 模板编排器”，不是文本生成器。

---

### 4.3 插件三：`并行生成`

当前状态：

- 是当前前端主链路中的真实第三段
- 但它不是只调用一次，而是每章调用一次

#### 4.3.1 它的职责

在“用户问题 + 场景结果 + 图谱素材 + 当前章节模板”四类输入的共同约束下，生成该章节的正式预案文本。

#### 4.3.2 它的内部节点工作方式

这个插件当前内部结构很简单，是典型的“输入 -> 单个 chatNode -> 输出”：

1. `pluginInput`
2. `chatNode`
   - 节点名：`AI 对话`
   - 作用：根据系统提示词生成本章正文
3. `pluginOutput`

#### 4.3.3 它的输入契约

当前固定输入变量是 4 个：

```json
{
  "用户问题": "用户原问题",
  "故障与场景提取结果": "结构化场景 JSON 字符串",
  "图谱检索方案素材": "图谱素材字符串",
  "模板": "当前章节模板文本"
}
```

#### 4.3.4 它的模型与生成规则

根据仓库里保存的已发布插件定义，这个插件当前的核心配置是：

- 模型：`deepseek-v4-flash`
- 主要要求：
  - 严格按当前传入章节生成
  - 不能补写前后章节
  - 保留 `[KG] / [GEN] / [FIX]` 来源标记
  - 多故障场景下要做融合表达，不能逐故障平铺

#### 4.3.5 它为什么叫“并行生成”

不是插件内部并行，而是 Python 流水线在外部并发调用它。

也就是说：

- 第 1 章是一次插件调用
- 第 2 章是一次插件调用
- 第 3 章又是一次插件调用

这些调用由 `ThreadPoolExecutor` 并行发起，因此前端可以一边收到第 2 章的文本，一边第 4 章还在生成中。

#### 4.3.6 它的输出

插件输出键是：

- `template`

脚本会优先从 `pluginOutput.template` 中取正文；如果没有，再兜底取 `choices[0].message.content`。

### 4.4 扩展插件：`多故障基本信息获取`

当前状态：

- 脚本已支持
- 当前实际部署前端链路已经接通

触发条件是：

- 前端开启 `多故障检索`
- 部署代理 `docker/frontend-proxy/server.py` 把它翻译成 Python 参数 `--multi-fault`

#### 4.4.1 它的职责

与单故障版 `基本信息获取` 类似，但目标不是只提取一个故障，而是输出：

- `故障二级节点` 数组
- `主故障二级节点`
- `设备表`

也就是为“复合故障”场景准备一个多故障版本的场景结构。

#### 4.4.2 它和单故障版的关键差异

单故障版的 `图谱检索方案素材` 可以直接在插件内部就完成。

多故障版不是这样。

它更像是：

1. 先识别设备表
2. 再识别多个故障节点和主故障
3. 把“逐故障查询”交给下一个专用插件并行完成

所以它更偏“多故障场景解析器”。

---

### 4.5 扩展插件：`parallel_database_query_plugin`

当前状态：

- 只在脚本的多故障分支里使用
- 当前实际部署前端在开启多故障时会触发

#### 4.5.1 它的职责

给定：

- `设备表`
- `当前查询的二级故障`

返回该故障对应的图谱检索文本。

#### 4.5.2 它在系统中的位置

这个插件不是最终生成插件，也不是模板插件。

它是多故障模式下的“逐故障图谱检索器”。

脚本会对 `故障二级节点[]` 中的每一个故障并行调用它，然后把结果聚合成：

```json
{
  "设备表": "...",
  "故障二级节点": ["A", "B"],
  "主故障二级节点": "A",
  "逐故障图谱检索": {
    "A": "...",
    "B": "..."
  }
}
```

随后再把这份聚合结果交给 `并行生成`。

#### 4.5.3 它的价值

这一步把“多故障支持”从一句 prompt 要求，变成了真正的数据层并行查询。

也就是说，多故障不是在最后生成时才想起来，而是在图谱检索阶段就已经被拆解和聚合。

## 5. 从用户输入到最终预案输出的完整环节图

如果按当前已经落地的真实前端部署链路来画，可以写成：

```text
用户输入问题
  -> 前端 /plan 页面
  -> runPipelineStream()
  -> /api/pipeline/run
  -> docker/frontend-proxy/server.py
  -> python3 scripts/run_parallel_generation_pipeline.py
  -> FastGPT 插件：基本信息获取
     -> 设备识别
     -> 故障类型分析
     -> 图谱查询
     -> 图谱素材清洗
  -> FastGPT 插件：模板切片
     -> 读取 llmkg_templates
     -> 组装 chapters[]
  -> 对每个 chapter 并发调用 FastGPT 插件：并行生成
     -> chatNode 生成章节正文
  -> 若开启案例搜索：并发调用知识库案例检索插件
  -> 若开启多故障：主流水线切换到多故障基本信息获取 + 逐故障图谱聚合
  -> pipeline_result.json / pipeline_result.md
  -> SSE 回推前端
  -> 前端逐章渲染
  -> 用户看到完整预案
```

如果把多故障专门展开写，可以细化成：

```text
用户输入问题
  -> 多故障基本信息获取
  -> parallel_database_query_plugin(逐故障并发)
  -> 多故障图谱素材聚合
  -> 模板切片
  -> 并行生成(逐章并发)
  -> 最终预案
```

## 6. 当前实现里最关键的几个事实

### 6.1 真正控制生成质量的不是单个插件，而是三层耦合

当前结果质量主要由三部分共同决定：

1. `基本信息获取` 是否识别对了设备和故障
2. `模板切片` 是否把章节边界和写作要求组织得足够清楚
3. `并行生成` 是否在章节边界内正确融合 `KG / GEN / FIX`

任何一层偏掉，最终预案都会偏。

### 6.2 当前“案例搜索”是由部署代理层并发实现的

案例搜索不在 `run_parallel_generation_pipeline.py` 主脚本内部，而是在 `docker/frontend-proxy/server.py` 里与主流水线并发编排。

也就是说：

- 主脚本负责 `基本信息获取 -> 模板切片 -> 并行生成`
- 部署代理负责额外并发 `run_case_search(...)`

### 6.3 当前“多故障检索”已在部署代理层打通

多故障开关并不是由前端自己直接控制脚本，而是：

- 前端发送 `enableMultiFaultSearch`
- 部署代理读取该字段
- 部署代理追加 `--multi-fault`
- 主脚本切换到多故障链路

### 6.4 前端不是直接调用 FastGPT，而是“代理编排 + 脚本编排 + 插件调用”

这点很重要。

当前系统的真实结构不是：

```text
前端 -> FastGPT
```

而是：

```text
前端 -> 部署代理 -> Python 编排脚本 / 案例检索插件 -> 多个 FastGPT 插件
```

所以前端只是交互壳，真正的工作流编排权在 Python 脚本里。
