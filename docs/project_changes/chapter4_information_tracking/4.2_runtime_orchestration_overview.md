# 4.2 运行时编排层设计总述信息

本文档整理第4章“运行时编排层设计总述”所需事实材料。运行时编排层位于服务接入层与数据知识层之间，负责把用户输入的问题组织为可执行的生成流程，并将 FastGPT 插件、图谱查询、章节并发生成和结果落盘串联为一条完整链路。

## 信息来源

- `scripts/run_parallel_generation_pipeline.py`
- `docker/frontend-proxy/services/pipeline_service.py`
- `docker/frontend-proxy/server.py`

## 层级定位

本系统的运行时编排层不是单一工作流平台，也不是单一后端脚本，而是由“Python 流水线 + FastGPT 插件工作流”共同构成的混合式编排单元。其中，Python 流水线负责全局阶段调度、串并行控制、流式事件输出和结果持久化，FastGPT 插件负责完成场景解析、模板切片、图谱素材获取和章节正文生成。

该层的核心目标不是直接生成整篇预案，而是将长文本生成任务拆解为若干中间状态明确、边界清晰且可追踪的子阶段，使系统能够同时满足场景理解、模板约束、并行生成和过程可观测等要求。

## 编排目标

运行时编排层主要承担以下任务：

| 任务 | 说明 |
|---|---|
| 用户问题结构化 | 将自然语言问题转换为设备空间、故障类型、事件场景和图谱素材。 |
| 生成边界控制 | 在进入正文生成前拦截无关输入、不支持设备或设备故障不兼容问题。 |
| 模板任务拆解 | 将整份预案模板拆分为多个章节级生成任务。 |
| 章节并发生成 | 对一级章节并行调用生成插件，提高长文本生成效率。 |
| 流式过程回传 | 将阶段状态和章节增量文本实时推送给前端。 |
| 结果归档落盘 | 保存 JSON 与 Markdown 结果，支撑复盘、评估和实验分析。 |

## 完整阶段链路

从用户点击“运行流水线”到结果落盘，当前系统的运行时编排过程可概括为 6 个阶段。

| 阶段 | 执行主体 | 主要输入 | 主要输出 | 性质 |
|---|---|---|---|---|
| 1. 请求启动 | 服务接入层代理 | `question`、`stream`、`enableCaseSearch`、`enableMultiFaultSearch` | 子进程启动参数、运行目录 | 串行 |
| 2. 基本信息获取 | Python 流水线 + FastGPT 基本信息插件 | `用户问题` | 场景结构、图谱素材、设备空间、边界判定 | 串行；多故障模式下内部并行 |
| 3. 多故障图谱聚合 | Python 流水线 + 多故障查询插件 | `设备表`、各二级故障 | 聚合后的逐故障图谱素材 | 条件并行 |
| 4. 模板切片 | Python 流水线 + FastGPT 模板切片插件 | 当前启用模板 | 章节数组、章节模板文本 | 串行 |
| 5. 章节并行生成 | Python 流水线 + FastGPT 并行生成插件 | 用户问题、场景结构、图谱素材、单章模板 | 各章节正文与增量文本 | 并行 |
| 6. 结果整理与落盘 | Python 流水线 | 中间响应与章节结果 | `pipeline_result.json`、`pipeline_result.md`、最终完成事件 | 串行 |

其中，阶段 2 至阶段 6 构成预案生成主链路，阶段 1 由服务接入层负责将外部请求转换为可执行的运行时任务。

## 分阶段说明

### 1. 请求启动

用户在前端页面输入故障问题后，请求首先进入服务接入层 `/api/pipeline/run` 接口。代理层根据请求体中的 `enableMultiFaultSearch` 决定是否向流水线脚本追加 `--multi-fault` 参数，根据 `stream` 决定采用同步返回模式还是 SSE 流式模式，并为本次运行创建独立的结果目录。

这一阶段不负责生成内容，而是完成运行上下文初始化，包括脚本启动、参数装配和结果目录创建。

### 2. 基本信息获取

基本信息阶段是整个生成链路的前置入口。单故障模式下，流水线调用 `基本信息获取` 插件；多故障模式下，流水线会并发调用单故障 `基本信息获取` 插件和 `多故障基本信息获取` 插件。

该阶段输出的核心结果包括：

| 字段 | 作用 |
|---|---|
| `用户问题` | 保留原始输入语义。 |
| `故障与场景提取结果` | 提供设备对象、故障二级节点、事件场景和处置要求等结构化场景约束。 |
| `图谱检索方案素材` | 提供故障原因、现象、后果、风险、措施和资源等知识图谱素材。 |
| `知识库名` 或 `设备表` | 标识当前命中的设备图空间，供案例检索或多故障查询使用。 |
| `边界判定结果` 与 `边界判定信息` | 决定流程是否可以继续进入模板切片和正文生成。 |

若边界判定结果不是 `ok`，流水线会直接发送 `pipeline_error` 并终止，不再继续后续阶段。因此，这一阶段同时承担场景理解和输入边界控制两项职责。

### 3. 多故障图谱聚合

该阶段只在多故障模式下执行。流水线首先从多故障基本信息结果中提取多个二级故障节点，然后对每个故障并发调用 `二级故障信息查询` 插件，分别获取该故障的图谱检索结果，最终聚合为统一的多故障素材结构。

聚合后结果通常包含：设备图空间、主故障二级节点、命中的全部故障二级节点以及“逐故障图谱检索”字典。随后，流水线再将该聚合结构转换为面向生成模型的文本材料，替换单故障模式下的普通图谱素材字段。

### 4. 模板切片

模板切片阶段调用 `模板切片` 插件，从 `llmkg_templates` 中读取当前启用模板，并将整份预案拆解为章节级任务数组。每个章节任务至少包含章节编号、标题、小节数量和当前章节展开后的模板文本。

该阶段的意义在于把“整篇预案生成”转换为“多个可独立执行的章节生成任务”。因此，模板切片是后续章节并发执行的结构基础。

### 5. 章节并行生成

模板切片完成后，流水线以一级章节为粒度建立线程池，并对每一章分别调用一次 `并行生成` 插件。每章调用共享同一份用户问题、场景结构和图谱素材，但使用不同的单章模板文本。

每个章节任务均会：

1. 构造标准化输入变量。
2. 以流式方式调用生成插件。
3. 将插件返回的文本增量转为 `chapter_chunk` 事件。
4. 对章节标题进行边界校验，不匹配时触发重试。
5. 完成后输出章节正文、耗时、状态和原始响应。

这一阶段是系统中最主要的并行计算阶段，也是运行时编排层区别于单次整篇生成的重要特征。

### 6. 结果整理与落盘

全部章节执行结束后，流水线会按章节编号对结果重新排序，并写入两类结果文件：

| 文件 | 作用 |
|---|---|
| `pipeline_result.json` | 保存问题、各阶段原始响应、标准化字段和章节结果，便于程序读取与实验分析。 |
| `pipeline_result.md` | 保存面向人工复核的 Markdown 结果，便于查看模板文本、章节正文、耗时和错误信息。 |

在流式模式下，流水线还会发送 `pipeline_done` 事件，将适合前端展示的简化结果结构返回给浏览器。至此，一次完整运行结束。

## 串行与并行边界

运行时编排层采用“前置串行、主体并行、收尾串行”的组织方式。

| 部分 | 设计方式 | 原因 |
|---|---|---|
| 基本信息获取 | 串行为主 | 后续所有阶段都依赖场景结构和图谱素材，必须先完成。 |
| 多故障基本信息双调用 | 小规模并行 | 为复用单故障边界判定能力，同时保留多故障解析结果。 |
| 多故障逐故障图谱查询 | 并行 | 各二级故障之间相互独立，适合并发聚合。 |
| 模板切片 | 串行 | 章节任务数组必须先构建，后续才能调度章节生成。 |
| 章节生成 | 大规模并行 | 各一级章节共享输入但互不依赖，适合并发调用生成插件。 |
| 最终排序与落盘 | 串行 | 需要保证最终文件结构稳定且按章节顺序输出。 |

这种串并行划分遵循两个原则：一是凡属于后续阶段公共依赖的信息先串行完成；二是凡能按故障或章节拆分、且彼此无直接依赖的任务尽量并行执行，以缩短整篇预案生成时间。

## 阶段间数据衔接

运行时编排层的关键设计不只是阶段拆分，还包括阶段之间使用统一中间表示完成数据衔接。

### 基本信息阶段到模板切片阶段

基本信息阶段产出的结构化场景结果和图谱素材不会直接写入模板，而是作为后续章节生成的共享上下文暂存于流水线内存结构中。模板切片阶段只负责生成章节任务数组，两者在逻辑上前后衔接，但数据上相互独立。

### 模板切片阶段到章节生成阶段

章节生成阶段对每个章节都使用统一输入结构：

```json
{
  "用户问题": "用户原问题",
  "故障与场景提取结果": "结构化场景结果",
  "图谱检索方案素材": "图谱素材",
  "模板": "当前章节模板文本"
}
```

这意味着所有章节共享相同的故障背景和知识素材，但仅在模板字段上存在差异，从而既保证了全文语义一致性，又实现了章节任务的独立并发执行。

### 中间结果到最终输出阶段

每个阶段的原始响应和标准化结果都会保存在流水线内部结果对象中。最终落盘时，这些信息被统一写入结构化 JSON；章节模板、生成正文、状态和耗时则同步写入 Markdown 文件，以便论文实验、误差分析和人工复核使用。

## 流程图文字版

若将运行时编排层绘制为流程图，其主流程可写为：

1. 前端提交生成请求到服务接入层。
2. 服务接入层创建运行目录并启动 Python 流水线。
3. 流水线进入基本信息阶段，调用基本信息插件。
4. 若边界判定失败，则返回终止信息并结束流程。
5. 若开启多故障模式，则并发完成多故障识别和逐故障图谱查询聚合。
6. 流水线调用模板切片插件，得到章节任务数组。
7. 流水线建立章节线程池，对各一级章节并发调用生成插件。
8. 各章节流式返回增量文本，并持续向前端发送章节事件。
9. 全部章节完成后，流水线按编号排序并写入 JSON 与 Markdown 结果文件。
10. 流水线发送最终完成事件，服务接入层结束 SSE 连接。

如果需要在论文中进一步压缩为一句话概括，可表述为：系统运行时编排层先完成场景解析与模板拆分，再以章节为粒度并发调用生成插件，最终对多阶段结果进行排序、落盘和流式回传。

## 与服务接入层的配合关系

运行时编排层与服务接入层之间采用“子进程 + 标准输出事件”的协作方式。服务接入层不直接控制各插件节点细节，而是负责启动流水线、转发事件和补充案例检索等外围能力；运行时编排层则专注于主生成链路的阶段调度和结果组织。该分工使系统既保留了部署代理层的统一入口优势，又避免将复杂的生成逻辑全部堆积在 HTTP 请求处理代码中。

## 论文可用表述

系统运行时编排层采用 Python 流水线与 FastGPT 插件协同的混合式控制方式。其运行过程并非单次整篇生成，而是首先通过基本信息插件将用户问题转换为结构化场景结果和图谱素材，在通过边界校验后调用模板切片插件将预案模板拆分为章节级任务，再以一级章节为粒度并发调用生成插件完成正文写作。对于多故障场景，系统在基本信息阶段额外引入多故障识别和逐故障图谱聚合分支。全部章节完成后，流水线对结果进行排序、落盘并以流式事件形式回传前端。该设计使生成流程具有明确的阶段边界、统一的中间表示和可追踪的运行证据，为系统的实时交互、实验评估和问题复盘提供了基础。
