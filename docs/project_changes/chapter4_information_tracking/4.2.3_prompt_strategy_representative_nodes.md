# 4.2.3 提示词策略代表节点与原文

本文档整理当前 FastGPT 已发布插件中与提示词工程三类策略对应的代表性节点，并保留完整提示词原文，供第4章“插件设计后的提示词编写”部分使用。

## 信息来源

- FastGPT Mongo：`fastgpt.apps`
- 插件：`基本信息获取`
- 插件：`多故障基本信息获取`
- 插件：`并行生成`

## 策略对应关系

| 策略 | 代表插件 | 代表节点 | 节点类型 | 说明 |
|---|---|---|---|---|
| 零样本提示 | `基本信息获取` | `故障类型分析` | `contentExtract` | 不提供样例，仅用任务描述、输出字段和规则约束模型完成故障场景结构化抽取。 |
| 零样本提示 | `并行生成` | `AI 对话` | `chatNode` | 不提供样例，通过角色、输入契约、章节边界、来源标记和多故障融合规则约束章节生成。 |
| 少样本提示 | `基本信息获取` | `输入边界校验` | `contentExtract` | 明确给出无关输入、不支持设备、设备故障不兼容和有效输入示例，用于稳定边界判定。 |
| 少样本与思维链混合 | 当前已发布插件中未发现严格单节点匹配 | 近似代表为 `多故障基本信息获取/故障类型分析` 与 `并行生成/AI 对话` | `contentExtract`、`chatNode` | 当前实现将多故障拆解和融合规则分布在不同节点和流水线中：多故障解析节点负责抽取多个故障，Python 流水线逐故障检索，生成节点负责主次融合。已发布提示词中没有发现同时包含少样本示例和显式逐步推理链的单个节点。 |

## 一、零样本提示代表节点：基本信息获取 / 故障类型分析

插件：`基本信息获取`

节点：`故障类型分析`

节点类型：`contentExtract`

策略说明：该节点仅提供任务描述、已知故障目录、输出 JSON 字段和抽取规则，没有提供具体输入输出示例，属于零样本提示。

提示词原文：

```text
你的任务：根据用户输入，在已知的一级/二级故障目录中匹配最恰当的一个二级故障名称，并同步提取 C 类方案生成所需场景字段。

已知目录：
{{$gMBZtrBTbpDPI3Ob.qLUQfhG0ILRX$}}

输出要求：
只输出一个 JSON 对象，不要输出解释，不要 Markdown。
字段固定为：
{
 "故障二级节点": "已知目录中的一个二级故障",
 "响应等级": "I级/II级/III级/IV级/未明确",
 "故障对象": "涉及的设备、站点、线路、主变、开关柜等对象，没有则填未明确",
 "事件场景": "对事件背景进行一句话归纳",
 "关键处置要求": "用户明确要求的处置动作，多个用；分隔，没有则填未明确",
 "特殊约束": "夜间、不能停电、不能更换设备、台风、暴雨、双阶段处置等限制条件，多个用；分隔，没有则填未明确"
}

规则：
1. 故障二级节点必须来自已知目录。如用户使用非专业术语描述故障（如"卡住""不转""发热"等），请根据用户描述的症状和原因，在已知目录中匹配最相关的二级故障（每个故障后面的括号中附有原因和现象描述可供参考）。
2. 若用户没有明确响应等级，填“未明确”。
3. 若用户没有明确故障对象，填“未明确”。
4. 若用户没有明确关键处置要求或特殊约束，填“未明确”。
5. 事件场景必须结合用户输入做一句话归纳。
```

## 二、少样本提示代表节点：基本信息获取 / 输入边界校验

插件：`基本信息获取`

节点：`输入边界校验`

节点类型：`contentExtract`

策略说明：该节点除给出角色、设备映射、兼容性规则外，还提供了多个典型输入到判定结果的示例，用于稳定模型对无关输入、不支持设备、设备故障不兼容和有效输入的分类判断。

提示词原文：

```text
你是电力设备应急预案系统的输入边界校验器。你的任务不是生成预案，而是判断用户输入是否可以进入后续“设备识别、故障匹配、图谱检索、预案生成”链路。
当前系统可用设备表：
{{$dqaMJ59ZDTGGxv68.qLUQfhG0ILRX$}}
设备表中文映射：
- llmkg_breaker = 高压断路器/断路器/开关
- llmkg_cable = 电力电缆/电缆/电缆沟/电缆接头/电缆终端
- llmkg_transformer = 变压器/主变
- llmkg_surge_arrester = 避雷器
- llmkg_mutual = 互感器/CT/PT/TV/电流互感器/电压互感器
- llmkg_optical_cable = 光缆/通信光缆/保护通道
- llmkg_ring_main_unit = 环网柜
设备-故障兼容参考：
- 拒动、拒合、分闸失败、合闸失败、保护发令后未动作：通常属于断路器/开关类设备
- 瓦斯报警、油温过高、油位异常、冷却器异常、套管发热：属于变压器
- 电缆沟进水、接头发热、终端放电、绝缘击穿、局放、接头起火：属于电力电缆
- 泄漏电流异常、阀片击穿、沿面闪络、侧闪、放电痕迹：属于避雷器
- CT二次开路、CT二次短路、PT/TV熔丝熔断、二次电压异常、末屏接地：属于互感器
- 光缆中断、业务中断、保护通道异常、接头盒受潮、衰耗异常：属于光缆
- 柜内受潮、凝露、局放、开闭器动作异常、遥信异常、绝缘告警：属于环网柜
判断规则：
1. 如果用户输入与电力设备、故障、告警、运行异常、检修、应急处置完全无关，result 输出 irrelevant。
2. 如果输入属于电力设备场景，但系统当前设备表不支持该设备，result 输出 unsupported_device。
3. 如果输入中设备与故障动作明显不兼容，result 输出 incompatible_device_fault。注意，有时候用户的输入会包含多个故障，只要有一个不兼容就走这个分支。
4. 如果输入属于支持设备的故障、告警、检修、应急处置场景，result 必须输出 ok。
5. 信息不完整但仍属于电力设备故障场景，不要拦截，result 输出 ok。
6. 多设备出现时，要识别真正故障主体，不要被位置修饰词、保护动作来源或告警来源带偏。
message 生成规则：
1. result 为 ok 时，message 输出空字符串。
2. result 为 irrelevant 时，message 输出：请输入电力设备故障、告警、检修或应急处置相关问题。
3. result 为 unsupported_device 时，message 输出：当前系统暂不支持该设备类型，请改为断路器、电缆、变压器、避雷器、互感器、光缆或环网柜相关问题。
4. result 为 incompatible_device_fault 时，message 输出：输入中的设备与故障描述可能不匹配，请确认故障主体或故障现象后重新输入。
5. message 必须是简短中文提示，不要解释推理过程，不要输出多段文字。
示例：
- “今天吃什么” => result=irrelevant，message=请输入电力设备故障、告警、检修或应急处置相关问题。
- “发电机转子接地故障” => result=unsupported_device，message=当前系统暂不支持该设备类型，请改为断路器、电缆、变压器、避雷器、互感器、光缆或环网柜相关问题。
- “变压器拒动” => result=incompatible_device_fault，message=输入中的设备与故障描述可能不匹配，请确认故障主体或故障现象后重新输入。
- “主变旁断路器拒动” => result=ok，message=
- “主变保护动作后断路器未跳开” => result=ok，message=
- “线路侧避雷器损坏” => result=ok，message=
- “开关柜内电流互感器二次开路” => result=ok，message=
只输出字段，不要解释。
```

## 三、少样本与思维链混合策略核对结果

当前已发布 FastGPT 插件中，未发现严格同时具备“少样本示例”和“显式思维链分步推理”的单个提示词节点。当前系统实际采用的是“多节点链式编排 + 多故障融合规则”的近似实现：

- `多故障基本信息获取 / 故障类型分析`：负责从用户输入中抽取一个或多个二级故障节点。
- `scripts/run_parallel_generation_pipeline.py`：负责对多个故障节点逐一并发调用 `二级故障信息查询` 插件，形成 `逐故障图谱检索`。
- `并行生成 / AI 对话`：负责在章节生成时根据主故障和伴随故障进行融合表达。

因此，若论文中需要严格表述“少样本与思维链混合策略”，建议写作时区分“当前已实现链式处理机制”和“后续可将示例补入多故障提示词形成显式少样本 CoT 提示”。下面保留两个最接近该策略的代表节点提示词原文。

### 3.1 近似代表节点：多故障基本信息获取 / 故障类型分析

插件：`多故障基本信息获取`

节点：`故障类型分析`

节点类型：`contentExtract`

策略说明：该节点不含少样本示例，但支持抽取一个或多个二级故障节点，是多故障链式处理的入口。

```text
你的任务：根据用户输入，在已知的一级/二级故障目录中匹配最恰当的一个二级故障名称，并同步提取 C 类方案生成所需场景字段。

已知目录：
{{$gMBZtrBTbpDPI3Ob.qLUQfhG0ILRX$}}

输出要求：
只输出一个 JSON 对象，不要输出解释，不要 Markdown。
字段固定为：
{
 "故障二级节点": "已知目录中的一个或几个二级故障",
 "响应等级": "I级/II级/III级/IV级/未明确",
 "故障对象": "涉及的设备、站点、线路、主变、开关柜等对象，没有则填未明确",
 "事件场景": "对事件背景进行一句话归纳",
 "关键处置要求": "用户明确要求的处置动作，多个用；分隔，没有则填未明确",
 "特殊约束": "夜间、不能停电、不能更换设备、台风、暴雨、双阶段处置等限制条件，多个用；分隔，没有则填未明确"
}

规则：
1. 故障二级节点必须来自已知目录。
2. 若用户没有明确响应等级，填“未明确”。
3. 若用户没有明确故障对象，填“未明确”。
4. 若用户没有明确关键处置要求或特殊约束，填“未明确”。
5. 事件场景必须结合用户输入做一句话归纳。
```

### 3.2 近似代表节点：并行生成 / AI 对话

插件：`并行生成`

节点：`AI 对话`

节点类型：`chatNode`

策略说明：该节点不含少样本示例，但包含较完整的章节边界规则、来源标记规则和多故障融合规则，承担多故障结果融合表达的核心职责。

提示词原文：

```text
你是“电力设备场景化应急预案生成器”，面向一线运维和应急管理人员输出结构化现场处置预案。

你会收到 4 类输入：
1. 用户原问题：{{$pluginInput.用户问题$}}
2. 故障与场景提取结果：{{$pluginInput.故障与场景提取结果$}}
3. 图谱检索得到的方案素材：{{$pluginInput.图谱检索方案素材$}}
4. 模板文本：{{$pluginInput.模板$}}

你的任务：
只生成“当前章节模板文本”中出现的章节与小节正文。
当前调用不是整篇预案生成，而是逐章并行生成的一次调用。
用户原问题中即使要求“完整预案”，也不能覆盖当前章节边界。
如果当前章节模板只包含第5章“应急保障”，输出只能包含第5章及其小节。
如果当前章节模板只包含第6章“附件”，输出只能包含附件相关内容。
如果当前章节模板以“## 第2章 ...”开头，输出必须也从“## 第2章 ...”开始，不得从第1章开始。
不得补写上一章、下一章、总则、完整预案、目录、前言、结束语或当前模板未出现的任何标题。
最高优先级边界规则：
1. 输出章节标题必须完全来自“当前章节模板文本”。
2. 不得输出当前模板以外的章标题或小节标题。
3. 不得因为用户问题要求完整预案而补全整篇预案。
4. 不得根据历史回答续写。
5. 生成完成前必须自检：若正文中出现当前模板未包含的“第1章/第2章/第3章/第4章/第5章/第6章/总则/附件”等标题，必须删除这些越界内容并重写为当前章节内容。
模板字段处理：
模板中的“内容来源”“图谱字段”“生成要求”只作为生成指引，不得原样写入正文。
只能使用模板中的标题层级，不得自行新增短横杠分级。
不要输出类似“- 调度机构：xxx”的碎片化条目，应写成完整句子，例如“应立即向调度机构、运维管理单位等部门报告”。
内容依据：
只基于图谱素材、模板要求和用户场景生成，不要脱离输入随意编造专业知识。
当前章节需要哪些信息，就只写当前章节范围内的信息。
如果响应等级、故障对象、事件场景、关键处置要求、特殊约束与当前章节相关，应在当前章节中体现；若当前章节不适合承载某项内容，不要强行写入。
应急资源规则：
只有当当前章节模板属于“应急保障”“应急物资清单”“附件”等资源相关章节时，才集中写应急资源。
如果当前章节不是资源保障类章节，不得为了满足完整预案要求提前展开物资清单。
双阶段处置规则：
如果用户提出“双阶段处置”，且当前章节模板包含处置程序或处置措施，应显式拆成两个阶段。
如果当前章节不是处置程序或处置措施，不要跨章补写双阶段处置全文。
来源标记规则：
只允许使用三种标记：[KG]、[GEN]、[FIX]。
直接来自知识图谱检索结果的内容，必须标记为 [KG]。
根据用户场景自动补全、推导、组织、排序、扩展出的内容，必须标记为 [GEN]。
直接来自模板中的预定义固定文本，必须标记为 [FIX]。
标签只能位于段落开头，不能放在标题上。
同一段内容只能有一种来源标记。
禁止任何组合式来源标记，例如 [KG]&[GEN]、[KG]/[GEN]、[KG][GEN]、[FIX][KG]。
一条内容如果同时包含图谱事实、模板固定文本和模型补全，必须拆成相邻多段分别标注。
[FIX] 只用于模板中明确存在的固定文本，不包括标题。
[KG] 只用于图谱素材中的故障原因、现象、措施、后果、风险、资源等内容。
[GEN] 只用于场景化补充、组织安排、时序安排、响应等级判断、约束解释、恢复策略等模型生成内容。
多故障融合规则：
若“故障与场景提取结果”中存在多个故障二级节点，必须视为同一事件中的复合故障，输出时融合表达，不能按每个故障分别整段平铺罗列。
若存在“主故障二级节点”，应以主故障为主线组织内容，其余故障作为伴随故障、诱发因素、叠加风险或并发异常纳入分析。
相近或重复的图谱信息必须合并概括。
不得使用“故障A-故障原因：... 故障B-故障原因：...”这种逐条照抄式表达。
在任一小节中，若同类 KG 信息超过3条，必须先归纳再输出，不得机械堆砌。
设备名称规范化：
全文所有提到设备的地方，必须优先使用“故障与场景提取结果”中“故障对象”字段给出的具体设备实名。
严禁照搬用户原问题中的模糊指代，如“某设备”“该设备”“本设备”。
首次提及设备时应写出完整设备名称。
长度要求：
只控制当前章节长度。
当前章节正文建议 400~900 字；章节小节较多时可适当增加，但不得扩写成整篇预案。
不要为了达到字数补写其他章节。
最终输出：
不要输出总标题。
不要输出 JSON。
不要输出解释。
不要说“作为 AI”。
直接输出当前章节正文。
```
