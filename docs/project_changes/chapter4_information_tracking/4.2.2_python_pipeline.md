# 4.2.2 Python 流水线设计信息

本文档整理 `scripts/run_parallel_generation_pipeline.py` 的运行机制，供第4章“Python 流水线设计”写作使用。该脚本是当前部署链路中的运行时编排核心，负责串联 FastGPT 插件、控制章节并发生成、向代理层输出事件并持久化结果。

## 执行阶段

当前脚本的主流程是“串行前处理 + 并行章节生成 + 结果落盘”。阶段如下：

| 阶段 | 执行方式 | 调用对象 | 输入 | 输出 |
|---|---|---|---|---|
| 基本信息获取 | 单故障串行；多故障模式下并行调用单故障和多故障插件 | `基本信息获取`、`多故障基本信息获取` | `用户问题` | 单故障结果用于边界判定，多故障结果用于后续场景结构。 |
| 多故障图谱聚合 | 条件执行、并行 | `二级故障信息查询`，代码 key 文件名保留 `parallel_database_query_plugin` | `设备表`、每个二级故障 | 聚合后的逐故障图谱素材。 |
| 模板切片 | 串行 | `模板切片` 插件 | 无显式业务变量 | 模板 ID、模板名称、版本、章节数组。 |
| 章节生成 | 并行 | `并行生成` 插件 | 用户问题、场景结果、图谱素材、当前章节模板 | 各章节正文、耗时、状态和错误信息。 |
| 结果整理 | 串行 | 本地脚本 | 各阶段中间结果 | `pipeline_result.json`、`pipeline_result.md`、最终事件。 |

| `extract_plugin_output()` | 从 FastGPT `responseData` 中提取 `pluginOutput`。 |
| `extract_boundary_failure()` | 识别输入越界、设备不支持或设备故障不匹配等边界失败。 |
| `extract_basic_fields()` | 将基本信息插件输出标准化为流水线统一字段。 |

| `write_outputs()` | 将完整流水线结果写入 JSON 和 Markdown 文件。 |

## 串并行设计

基本信息获取和模板切片必须在章节生成前完成，因为后续章节生成依赖场景字段、图谱素材和章节模板。单故障模式下，系统只调用 `基本信息获取` 插件；多故障模式下，系统会同时调用单故障 `基本信息获取` 插件和 `多故障基本信息获取` 插件，先使用单故障插件结果进行边界判定，若单故障边界判定终止，则直接返回单故障边界信息；若单故障判定通过，则继续使用多故障插件结果进入逐故障图谱聚合和章节生成。章节生成阶段则天然可以并行，因为每个一级章节只需要共享的基本信息和自身模板文本，不依赖其他章节输出。

并发实现方式为 `ThreadPoolExecutor`：

- 多故障图谱查询使用 `ThreadPoolExecutor(max_workers=min(max_workers, len(fault_nodes)))`。
- 多故障基本信息阶段使用 `ThreadPoolExecutor(max_workers=2)` 同时调用单故障和多故障基本信息插件，以复用单故障边界检查能力。
- 章节生成使用 `ThreadPoolExecutor(max_workers=min(args.max_workers, len(chapters)))`。
- 默认最大并发度为 `6`。
- 并发任务通过 `as_completed()` 回收，因此章节完成顺序可能与章节编号不同。
- 所有章节完成后，脚本再按章节编号排序，保证最终结果顺序稳定。

该设计兼顾了长文本生成效率和最终文档结构稳定性：前端可以实时收到任意章节的生成增量，最终落盘结果仍按模板章节顺序排列。

## 章节生成输入

每个章节任务会构造如下变量传给并行生成插件：

```json
{
  "用户问题": "用户原问题",
  "故障与场景提取结果": "基本信息阶段标准化后的场景结果",
  "图谱检索方案素材": "单故障图谱素材或多故障聚合素材",
  "模板": "当前章节模板文本"
}
```

对于多故障图谱素材，`build_generation_graph_material()` 会将聚合 JSON 转为面向生成模型的文本，包含设备表、主故障二级节点、命中故障二级节点和每个故障的图谱检索内容。

## 流式事件

开启 `--stream-events` 后，脚本向标准输出写出 JSON 行事件。部署代理读取这些事件并转发给前端。

| 事件 | 触发时机 | 主要数据 |
|---|---|---|
| `basic_info_started` | 开始调用基本信息插件 | `question` |
| `basic_info_done` | 基本信息阶段完成 | `elapsed_sec`、`basicInfo` |
| `template_split_started` | 开始调用模板切片插件 | 空对象 |
| `template_split_done` | 模板切片完成 | 模板信息、章节数组、章节待生成状态 |
| `parallel_generating_started` | 开始章节并行生成 | `chapterCount` |
| `chapter_started` | 某章节任务提交到线程池 | 章节编号、标题、小节数 |
| `chapter_chunk` | 某章节收到流式文本增量 | 章节编号、标题、文本片段 |
| `chapter_retry` | 某章节标题校验失败并准备重试 | 章节编号、尝试次数、错误信息 |
| `chapter_done` | 某章节生成完成 | 章节正文、模板文本、耗时、状态 |
| `chapter_error` | 某章节调用抛出异常 | 章节编号、标题、错误信息 |
| `pipeline_done` | 全部阶段完成 | 问题、基本信息、模板信息、完整章节结果 |
| `pipeline_error` | 边界判定失败或流程异常 | 错误信息或边界失败原因 |

其中 `chapter_chunk` 来自 FastGPT 流式返回中的 `answer` 事件，脚本将文本片段实时转为章节级增量事件。

## 重试与校验

`generate_one_chapter()` 会校验生成正文中的章节标题是否与当前模板章节匹配。默认最多尝试 3 次，可通过环境变量 `PIPELINE_CHAPTER_HEADING_MAX_ATTEMPTS` 调整。

校验逻辑包括：

- 从模板文本中提取期望的“第几章”。
- 从生成正文中提取实际输出的“第几章”。
- 支持阿拉伯数字和一至十的中文数字互相匹配。
- 若标题不匹配，发送 `chapter_retry` 事件并重试。
- 若全部尝试失败，该章节状态记为 `error`，正文置空，记录最后一次标题错误。

该机制用于约束模型不要生成错误章节或补全整篇预案。

## 结果落盘

流水线完成后会在输出根目录下创建时间戳子目录，时间戳格式为 `YYYYMMDD_HHMMSS`。每次运行写入两个文件：

| 文件 | 内容 |
|---|---|
| `pipeline_result.json` | 保存问题、基本信息阶段原始响应与标准化字段、模板切片响应与切片结果、各章节生成响应与正文。 |
| `pipeline_result.md` | 面向人工查看的 Markdown 结果，包含基本信息、模板切片结果、每章模板文本、输出正文、耗时、状态和错误。 |

`pipeline_result.json` 的顶层结构包括：

```json
{
  "question": "用户问题",
  "basic_info": {
    "elapsed_sec": 0,
    "fields": {},
    "response": {}
  },
  "template_splitter": {
    "elapsed_sec": 0,
    "split_result": {},
    "response": {}
  },
  "parallel_generations": []
}
```

最终推送给前端的 `pipeline_done` 事件则采用更适合展示的结构，包含 `basicInfo`、`templateSplit` 和 `chapters[]`。

## 异常处理

当前脚本主要处理三类异常：

| 类型 | 处理方式 |
|---|---|
| 输入边界失败 | 基本信息阶段返回非 `ok` 边界判定时，发送 `pipeline_error` 后提前结束；多故障模式下复用单故障基本信息插件的边界判定结果。 |
| 模板切片为空 | 若无法提取章节数组，抛出 `RuntimeError` 并发送 `pipeline_error`。 |
| 章节生成异常 | 单章 future 抛出异常时，发送 `chapter_error`，随后抛出异常并进入全局 `pipeline_error`。 |

需要注意的是，脚本当前具备章节标题校验重试，但没有实现完整的断点恢复机制。每次运行都会生成新的时间戳目录；历史结果可用于分析和复现，但失败后不会自动从上次成功章节继续执行。

## 论文可用表述

Python 流水线是系统运行时编排层的全局控制单元。它通过 FastGPT OpenAI 兼容接口依次调用基本信息获取、模板切片和并行生成插件，并在多故障场景下增加逐故障图谱查询分支。流水线首先串行完成场景解析和模板拆分，获得统一的场景字段、图谱素材和章节模板数组；随后以章节为任务粒度建立线程池，将不同章节的生成请求并发发送至生成插件，并通过流式事件向前端返回章节增量内容。所有章节完成后，流水线按章节编号重新排序，生成结构化 JSON 和 Markdown 结果文件。该设计将大模型长文本生成任务拆解为可并发、可追踪、可落盘的章节级任务，提高了生成效率，也为后续质量评估和误差分析保留了中间证据。
