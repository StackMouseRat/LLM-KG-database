## 主链路插件

### 基本信息获取

该插件是预案生成链路的第一段，主要完成“自然语言输入到结构化生成条件”的转换。它不是直接让模型生成预案，而是先识别设备空间和故障类型，再查询图谱并清洗为后续生成可使用的素材。

内部处理可概括为：

1. 接收用户问题。
2. 获取 Nebula 设备图空间列表。
3. 识别用户问题对应的设备空间。
4. 查询该设备空间下的一级、二级故障目录。
5. 通过内容抽取节点匹配最合适的二级故障，并提取响应等级、故障对象、事件场景、关键处置要求和特殊约束。
6. 围绕命中的二级故障查询原因、现象、措施、后果、风险和资源等下游节点。
7. 清洗图查询结果并输出图谱检索素材。

Python 流水线会将插件输出标准化为以下字段：

| 标准化字段 | 来源 | 用途 |
|---|---|---|
| `用户问题` | 插件输出或原始问题 | 保留用户原始语义。 |
| `故障与场景提取结果` | `故障类型分析` | 向生成阶段提供结构化场景约束。 |
| `图谱检索方案素材` | `图谱检索` | 向生成阶段提供知识图谱事实材料。 |
| `知识库名` | `知识库名` | 标识当前命中的设备图谱空间。 |
| `边界判定结果` | `边界判定结果`、`reason` 或 `result` | 判断输入是否可进入预案生成链路。 |
| `边界判定信息` | `边界判定信息` 或 `message` | 向前端返回不可生成原因。 |

边界判定结果如果为非 `ok`，流水线会直接返回 `pipeline_error`，不再进入模板切片和章节生成阶段。

### 模板切片

该插件是预案生成链路的第二段，本质上是模板读取与模板编排插件，不承担文本生成任务。它通过 HTTP 节点访问 `llmkg_templates` 图空间，读取当前模板版本、章节层级、内容来源、图谱字段、固定文本和生成要求，再由代码节点组装为章节级模板数组。

模板切片结果的核心结构如下：

| 字段 | 含义 |
|---|---|
| `template_id` | 当前模板标识。 |
| `template_name` | 当前模板名称。 |
| `current_version` | 当前启用模板版本。 |
| `chapters[]` | 按一级章节拆分后的生成任务数组。 |
| `chapter_no` | 章节编号。 |
| `title` | 章节标题。 |
| `section_count` | 当前章节包含的小节数量。 |
| `text` | 当前章节展开后的模板文本。 |

模板文本构造规则包括：一级标题写为 `## 第{section_no}章 {title}`，非一级标题按层级写为 Markdown 标题；如果章节配置存在内容来源、图谱字段、预定义文本或生成要求，则一并写入章节模板文本。该设计使生成插件收到的是已经展开的章节写作说明，而不是原始模板数据库记录。

### 并行生成

该插件是预案生成链路的第三段，负责生成单个章节的正式预案正文。插件内部是“输入节点 -> AI 对话节点 -> 输出节点”的生成结构，模型配置为 `deepseek-v4-flash`。

该插件固定接收四类输入：

```json
{
  "用户问题": "用户原问题",
  "故障与场景提取结果": "结构化场景结果字符串",
  "图谱检索方案素材": "图谱检索素材字符串",
  "模板": "当前章节模板文本"
}
```

生成规则重点包括：

- 严格依据当前传入章节模板生成，不能补写前一章、后一章或未出现在模板中的章节。
- 模板中的内容来源、图谱字段和生成要求只作为写作指引，不能原样暴露在正文中。
- 生成正文保留 `[KG]`、`[GEN]`、`[FIX]` 三类来源标记。
- 图谱直接提供的内容标记为 `[KG]`，模型基于场景组织和补全的内容标记为 `[GEN]`，模板固定文本标记为 `[FIX]`。
- 多故障场景下需要融合表达，不能对每个故障逐段平铺罗列。

“并行生成”的并行性不在 FastGPT 插件内部，而在 Python 流水线外部。流水线会为每个一级章节分别发起一次插件调用，并用线程池并发执行。

## 多故障扩展插件

### 多故障基本信息获取

当用户在前端开启“多故障检索”时，部署代理会将请求转换为 Python 脚本参数 `--multi-fault`。脚本随后同时调用单故障 `基本信息获取` 插件和 `多故障基本信息获取` 插件：单故障插件用于复用已有边界判定能力，若单故障边界判定终止，则直接返回单故障的边界信息；若单故障判定通过，则使用多故障插件输出继续执行逐故障图谱聚合和章节生成。

该插件内部结构与单故障版基本信息插件相近，均采用“设备表初始化 -> 获取全体设备表 -> 清洗设备表数据 -> 设备识别 -> 基础数据获取 -> 基础数据清洗 -> 故障类型分析 -> 插件输出”的节点链路。关键差异在于 `故障类型分析` 节点使用数组型提取字段，允许从同一用户问题中识别一个或多个二级故障。

该插件重点输出：

| 字段 | 含义 |
|---|---|
| `用户问题` | 原始用户问题。 |
| `故障类型分析` | JSON 字符串，包含多故障场景字段。 |
| `故障二级节点` | `故障类型分析` 内部字段，类型为数组，表示当前问题命中的一个或多个二级故障节点。 |
| `主故障二级节点` | Python 流水线标准化时补充的主线故障字段；如果插件输出未显式给出，则默认取数组首项。 |
| `设备表` | 当前故障所属设备图谱空间，例如 `llmkg_breaker`。 |

`故障类型分析` 节点的提取要求是：根据用户输入在已知一级/二级故障目录中匹配最恰当的一个或几个二级故障，并同步提取响应等级、故障对象、事件场景、关键处置要求和特殊约束。该节点使用模型 `Pro/deepseek-ai/DeepSeek-V3.2`，`history` 为 `0`。

与单故障插件相比，该插件主要承担多故障场景解析，不直接完成全部图谱素材聚合。

### 二级故障信息查询

该插件在代码和 key 文件中对应 `parallel_database_query_plugin`，但 FastGPT 中的应用名称为 `二级故障信息查询`。它只在多故障分支中使用。Python 脚本会对 `故障二级节点[]` 中的每个故障并行调用该插件，输入为 `设备表` 和 `当前查询的二级故障`，输出为该故障对应的图谱检索文本。

内部节点链路为：`插件输入 -> 下游节点获取 -> 下游节点数据清洗 -> 插件输出`。其中 `下游节点获取` 是 HTTP 请求节点，访问 Nebula HTTP 网关并围绕当前二级故障查询原因、现象、措施、后果、风险和资源等下游节点；`下游节点数据清洗` 是代码节点，负责把图查询结果整理为生成阶段可读的文本。

多故障查询结果会被聚合为如下结构，并作为新的 `图谱检索方案素材` 传给并行生成插件：

```json
{
  "设备表": "llmkg_breaker",
  "故障二级节点": ["故障A", "故障B"],
  "主故障二级节点": "故障A",
  "逐故障图谱检索": {
    "故障A": "...",
    "故障B": "..."
  }
}
```

## 论文可用表述

系统运行时编排层并未将预案生成封装为单次大模型调用，而是将 FastGPT 工作流拆分为场景解析、模板切片和章节生成三个插件。首先，基本信息获取插件完成设备空间识别、故障类型匹配、场景字段抽取和图谱素材查询；其次，模板切片插件从模板图空间读取当前启用模板，将整份预案结构拆分为章节级生成任务；最后，并行生成插件在“用户问题、场景结果、图谱素材和章节模板”的共同约束下生成单章正文。对于复合故障场景，系统进一步引入多故障基本信息插件和逐故障图谱查询插件，在生成前完成多故障识别和图谱素材聚合。上述设计使生成过程具备明确的中间状态、可追踪的数据来源和可扩展的插件边界。
