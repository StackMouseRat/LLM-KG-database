# 4.4 前端表现层设计信息

本文档整理第4章“前端表现层设计”所需事实材料。前端表现层并不直接承担知识图谱查询或大模型生成，而是负责把后端多阶段工作流转换为用户可观察、可控制、可复用的交互界面。其职责不仅包括预案生成页面本身，还覆盖登录鉴权、图谱溯源、质量评估、模板查看和对比实验等辅助页面。

## 层级定位

前端表现层位于系统最上层，其核心任务是围绕“预案生成全过程的可视化”组织页面，而不是简单提供一个文本输入框和结果展示框。当前实现采用 React + TypeScript 构建单页应用，通过统一的 `/api/` 接口访问服务接入层，并将后端返回的阶段事件、章节增量文本、图谱子图、模板配置和实验记录转化为多个业务页面。

从系统角色上看，前端主要承担四类职责：

| 职责 | 说明 |
|---|---|
| 交互入口 | 接收用户问题、控制案例搜索和多故障检索等运行参数。 |
| 过程可视化 | 展示基本信息获取、模板切片、并行生成、案例检索等阶段状态。 |
| 结果组织 | 以章节卡片、图谱视图、质量对比卡片和实验看板等形式展示输出。 |
| 状态复用 | 将一次生成结果在预案页、溯源页、质量页和实验页之间复用，减少重复输入。 |

## 技术栈与前端壳层

前端采用 React + TypeScript 实现，页面组件主要基于 Ant Design，构建工具为 Vite。应用入口位于 `frontend/src/App.tsx`，整体采用单页应用方式组织，不依赖 React Router，而是通过浏览器 `history` 和 `popstate` 事件手工维护当前路由状态。

### 全局页面壳层

`App.tsx` 负责构建整个前端壳层，主要包括：

| 模块 | 作用 |
|---|---|
| 顶部标题区 | 展示系统标题“电力设备智能预案生成系统”。 |
| 用户控制区 | 展示当前用户名、用户组、登出按钮和模型余额信息。 |
| 显示偏好区 | 提供“显示标签”“紧凑布局”“夜间模式”开关。 |
| 导航标签区 | 在预案生成、图谱溯源、格式优化与质量评估、模板查看、对比实验之间切换。 |
| 内容区 | 根据登录状态和当前路由渲染对应页面。 |

页面壳层还承担以下全局控制逻辑：

1. 首次加载时根据当前 URL 推断页面路由。
2. 调用 `useAuthSession()` 检查用户登录状态。
3. 在已登录状态下定时拉取模型供应商余额信息。
4. 将“显示标签”“紧凑布局”“夜间模式”持久化到浏览器 `localStorage`。
5. 将夜间模式同步为 `document.body` 的样式类。

### 全局显示偏好

前端在页面壳层维护三类全局显示偏好：

| 偏好项 | 存储键 | 作用 |
|---|---|---|
| `showModeTags` | `llmkg_mode_tags_visible_v1` | 控制是否显示 `[KG]`、`[GEN]`、`[FIX]` 来源标签。 |
| `showCompactLayout` | `llmkg_compact_layout_v1` | 控制章节卡片、模板卡片和评估卡片的紧凑排布方式。 |
| `darkMode` | `llmkg_dark_mode_v1` | 控制全局夜间模式及图谱页面配色。 |

这类偏好设置不影响后端生成逻辑，但会影响前端呈现方式，因此被归入表现层状态而非运行时参数。

## 登录与页面访问控制

当前前端并不是完全开放访问，而是先经过登录页，再进入业务页面。认证状态由 `useAuthSession()` 维护，登录接口由 `services/authApi.ts` 访问。

### 登录状态模型

前端认证状态分为三种：

| 状态 | 含义 |
|---|---|
| `checking` | 正在调用 `/api/auth/me` 检查已有会话。 |
| `authenticated` | 已登录，可访问业务页面。 |
| `unauthenticated` | 未登录，仅允许进入 `/login`。 |

### 登录流程

1. 页面加载时调用 `fetchCurrentUser()` 请求 `/api/auth/me`。
2. 若返回 401，则转到登录页。
3. 登录页 `LoginPage` 提供用户名和密码输入框，提交到 `/api/auth/login`。
4. 登录成功后，前端写入当前用户名与用户组，跳转到 `/plan`。
5. 登出时调用 `/api/auth/logout`，再将路由切回 `/login`。

### 用户组权限

前端区分 `admin` 和 `user` 两类用户组，并在页面中显式控制权限：

| 页面或功能 | 普通用户 | 管理员 |
|---|---|---|
| 预案生成 | 可用 | 可用 |
| 图谱溯源 | 可用 | 可用 |
| 质量评估查看 | 可用 | 可用 |
| 模板提示词编辑 | 不可用 | 可用 |
| 模板章节编辑 | 不可用 | 可用 |
| 模板恢复默认 | 不可用 | 可用 |

因此，前端不仅负责展示内容，也承担一部分基于用户组的交互约束。

## 页面路由与页面组成

当前页面路由配置位于 `frontend/src/app/routeConfig.ts`。

| 路由 | 页面 | 主要功能 |
|---|---|---|
| `/login` | 登录页 | 用户登录。 |
| `/plan` | 预案生成页 | 输入故障问题，运行流水线，查看章节结果和案例搜索。 |
| `/trace` | 图谱溯源页 | 根据最近一次生成结果查询并展示图谱子图。 |
| `/quality` | 格式优化与质量评估页 | 对章节原文执行优化和评估。 |
| `/template` | 模板查看页 | 查看并按章节编辑模板配置。 |
| `/experiment` | 对比实验页 | 运行实验、查看实验记录与评估结果。 |

其中，除登录页外，其余页面都依赖用户已通过认证。应用在渲染这些页面时使用 `React.lazy` 和 `Suspense` 进行懒加载，以减少初始加载开销。

## 预案生成页设计

预案生成页是前端表现层的核心页面，对应 `PlanPage`。该页面本身很薄，真正的业务逻辑主要集中在 `usePlanPipeline()` 状态钩子以及三个可视化组件中。

### 页面组成

预案生成页由以下三部分组成：

| 组件 | 位置 | 职责 |
|---|---|---|
| `PlanInputCard` | 顶部 | 输入问题、启动运行、控制案例搜索和多故障检索、显示阶段标签与摘要标签。 |
| `PlanChapterGrid` | 中部 | 以章节卡片方式展示模板文本、正文输出、耗时与状态。 |
| `CaseSearchPanel` | 底部 | 展示案例检索状态和命中的历史案例卡片。 |

这种布局体现了“输入控制区 -> 主生成结果区 -> 辅助案例区”的主次关系，便于用户在一个页面中观察整条链路。

### 预案生成页状态中心

`usePlanPipeline()` 是预案生成页的核心状态管理单元。它维护如下关键状态：

| 状态 | 含义 |
|---|---|
| `question` | 当前输入的问题文本。 |
| `pipeline` | 当前流水线结果对象，含基本信息、模板信息、章节和案例搜索结果。 |
| `stage` | 当前主阶段。 |
| `nodeStageLabel` | 面向用户显示的中文阶段提示。 |
| `loading` | 是否正在运行生成流程。 |
| `enableCaseSearch` | 是否开启案例搜索。 |
| `enableMultiFaultSearch` | 是否开启多故障检索。 |
| `savedFlag` | 最近一次结果是否已保存为本地快照。 |
| `terminationInfo` | 边界终止时的提示信息。 |
| `questionPopoverOpen` | 示例问题弹窗是否打开。 |

### 阶段模型

前端将运行过程抽象为以下阶段类型：

| 阶段 | 含义 |
|---|---|
| `idle` | 尚未开始。 |
| `basic_info` | 正在获取基本信息。 |
| `template_split` | 正在切分模板。 |
| `parallel_generating` | 正在并行生成章节。 |
| `case_search` | 正在并发检索案例。 |
| `done` | 生成完成。 |
| `terminated` | 因边界判定而提前终止。 |
| `error` | 流程执行失败。 |

这个阶段模型既用于顶部状态标签展示，也用于控制按钮、提示信息和各页面之间的行为。

### 输入控件与操作项

`PlanInputCard` 提供以下交互控件：

| 控件 | 作用 |
|---|---|
| 多行文本框 | 输入自然语言故障问题。 |
| `运行流水线` 按钮 | 启动流式生成。 |
| `填入示例` 按钮 | 打开示例问题弹窗。 |
| `复制全部` 按钮 | 将合并后的章节 Markdown 复制到剪贴板。 |
| `下载全部` 按钮 | 下载合并后的 Markdown 文件。 |
| `开启案例搜索` 复选框 | 控制服务接入层是否并发案例检索。 |
| `开启多故障检索` 复选框 | 控制后端是否切换到多故障生成链路。 |

此外，输入卡还显示一组动态标签，包括：

1. 当前主阶段标签。
2. 当前节点阶段提示标签。
3. 已保存标签。
4. 边界判定结果与边界信息标签。
5. 根据 `faultScene` 解析出的摘要标签，如故障二级节点、故障对象、模板名称和章节数量。

若本次请求被边界判定终止，页面还会显示 `Alert` 形式的终止提示框，而不是简单弹出报错。

### SSE 事件消费机制

前端预案生成页不使用轮询，也不等待整篇结果一次性返回，而是通过 `runPipelineStream()` 以 `Accept: text/event-stream` 发起请求，并自行解析 SSE 数据流。

`runPipelineStream()` 的处理过程包括：

1. 使用 `fetch('/api/pipeline/run')` 发起 POST 请求。
2. 将请求体扩展为 `{ ...payload, stream: true }`。
3. 通过 `response.body.getReader()` 逐块读取字节流。
4. 按 SSE 的 `event:` 和 `data:` 协议切分事件块。
5. 解析事件名称后，分别调用不同的页面状态处理回调。

### 事件到页面状态的映射

| 后端事件 | 前端处理 |
|---|---|
| `basic_info_started` | 将阶段置为 `basic_info`，提示“正在获取基本信息”。 |
| `basic_info_done` | 写入 `basicInfo`，包括用户问题、场景结果、图谱素材和边界判定信息。 |
| `template_split_started` | 将阶段置为 `template_split`。 |
| `template_split_done` | 写入模板元信息，初始化章节数组和章节卡片骨架。 |
| `parallel_generating_started` | 将阶段置为 `parallel_generating`。 |
| `chapter_started` | 将对应章节状态置为 `running`。 |
| `chapter_chunk` | 将增量文本追加到对应章节的 `outputText`。 |
| `chapter_done` | 为章节写入最终正文、耗时和完成状态。 |
| `case_search_started` | 初始化案例搜索状态为 `running`。 |
| `case_search_done` | 写入案例卡片并更新状态。 |
| `case_search_error` | 显示失败或跳过信息。 |
| `pipeline_done` | 保存完整结果快照并进入 `done` 状态。 |
| `pipeline_error` | 若带 `reason` 字段则进入 `terminated`，否则进入 `error`。 |

这种机制使前端能够把后端复杂的多阶段运行过程拆成一系列可感知的交互节点，而不是只展示一个全局 loading 状态。

### 章节卡片展示

`PlanChapterGrid` 采用网格卡片形式展示每个章节。每张卡片包含：

| 区域 | 内容 |
|---|---|
| 卡片标题 | 章节编号与标题。 |
| 右上标签 | 章节状态，如等待中、生成中、已完成、失败。 |
| 元信息区 | 耗时与小节数量。 |
| 折叠面板 | 当前章节模板文本。 |
| 正文区 | 生成结果，使用 `RichTextRenderer` 渲染。 |

章节卡片支持两种布局：

| 布局 | 表现 |
|---|---|
| 普通布局 | 按章节数自动铺开，适合横向比较少量章节。 |
| 紧凑布局 | 固定为 3 列，适合屏幕较窄或需要更密集排布时使用。 |

由于 `chapter_chunk` 事件会持续追加文本，因此章节卡片正文区实际上具备“边生成边显示”的流式输出效果。

### 案例检索面板

`CaseSearchPanel` 对案例搜索结果进行独立展示。其状态与主流水线并列，但由同一个 `pipeline` 对象保存。

案例面板可呈现以下几种状态：

| 状态 | 页面表现 |
|---|---|
| 未开启 | 显示“未开启案例搜索”。 |
| 等待识别设备 | 显示“等待开始案例检索”或“等待识别设备”。 |
| 运行中 | 显示“正在检索案例，请稍候”。 |
| 跳过 | 显示未命中设备知识库。 |
| 失败 | 显示失败原因。 |
| 完成 | 以卡片网格显示案例标题、知识库 ID、文档 ID、相关性和摘要。 |

这部分展示将辅助知识检索与主生成结果解耦，避免影响章节展示区域的可读性。

### 本地快照与结果导出

预案生成页会在成功收到 `pipeline_done` 后调用 `saveSnapshot()`，将最近一次生成结果保存到浏览器 `localStorage`，键名为 `llmkg_saved_plan_snapshot_v1`。页面重新打开后，`loadSavedSnapshot()` 会自动恢复该结果，并把阶段状态设置为 `done`。

这种本地持久化机制主要带来两个作用：

1. 避免页面刷新后丢失最近一次成功结果。
2. 为图谱溯源页和质量评估页提供稳定的上游数据来源。

## 图谱溯源页设计

图谱溯源页对应 `TraceGraphPage`，其输入不是新的用户问题，而是最近一次预案生成结果中的 `question`、`faultScene` 和 `graphMaterial`。这体现了页面之间的数据复用关系。

### 页面目标

该页面用于回答“当前预案是基于哪些图谱节点和关系得到的”这一问题，因此其核心是将文本化图谱素材转化为可视化子图。

### 页面状态

图谱页主要维护以下状态：

| 状态 | 作用 |
|---|---|
| `trace` | 当前图谱溯源结果对象。 |
| `loading` | 是否正在初次加载图谱。 |
| `refreshing` | 是否正在手动刷新。 |
| `errorText` | 查询失败时的错误提示。 |
| `selectedNodeId` | 当前选中的图节点。 |
| `shouldPlayAnimation` | 是否播放首次图谱展开动画。 |
| `graphReady` | 图可视化是否已完成渲染。 |

### 数据获取与缓存

图谱页通过 `fetchTraceSubgraph()` 请求 `/api/trace/subgraph`。若存在最近一次生成结果，则前端会将问题、场景结果和图谱素材一并传给后端，后端返回标准化的节点和边结构。

前端额外实现了图谱缓存与视口恢复逻辑，包括：

| 能力 | 作用 |
|---|---|
| 溯源缓存 | 按最近一次生成结果签名缓存图谱结果。 |
| 动画播放记录 | 防止同一份图谱重复播放首屏动画。 |
| 视口缓存 | 保存缩放、平移和旋转状态。 |
| 强制刷新 | 允许用户重新向后端请求最新子图。 |

### 可视化结构

页面采用左右分栏布局：

| 区域 | 内容 |
|---|---|
| 左侧“本次识别结果”卡片 | 展示设备根节点、主故障二级节点、命中二级故障、节点数、边数和命中节点数。 |
| 左侧“节点详情”卡片 | 展示当前点击节点的名称、类型和描述。 |
| 右侧图谱卡片 | 通过 G6 绘制图谱子图，并支持拖拽、缩放和节点点击。 |

图谱页面在表现层面进一步增强了系统的可解释性，使用户能够从生成结果跳转到知识来源结构。

## 格式优化与质量评估页设计

质量评估页对应 `QualityReviewPage`。该页面直接复用最近一次预案生成结果的章节数组，对每个章节执行二次优化和质量评价。

### 页面输入来源

质量页不会重新生成预案，而是从本地保存的流水线结果中读取：

| 数据 | 来源 |
|---|---|
| `chapters` | 最近一次成功生成的章节列表。 |
| `faultScene` | 最近一次基本信息结果。 |
| `graphMaterial` | 最近一次图谱素材。 |

因此，质量页本质上是建立在预案生成页结果之上的后处理界面。

### 页面结构

页面由两个部分组成：

| 组件 | 作用 |
|---|---|
| `QualityPromptPanel` | 展示并编辑优化提示词与评估提示词，支持批量触发。 |
| `QualityChapterReviewGrid` | 以章节为单位并排展示原文、优化后文本和评估结果。 |

### 章节评估卡片结构

每个章节在质量页会形成左右两张卡片：

| 卡片 | 内容 |
|---|---|
| 左卡 | 原文、原文评估结果或优化后评估结果，可在三种视图间切换。 |
| 右卡 | 优化推理过程与优化后正文，同时显示优化状态和评估状态标签。 |

用户可对单章执行：

1. 优化。
2. 评估原文。
3. 评估优化后。

页面也支持批量并发执行上述任务，默认并发上限为 6。

### 流式评估机制

质量页调用 `/api/quality/review` 时同样采用 SSE 流式模式。前端会解析如下事件：

| 事件 | 含义 |
|---|---|
| `quality_status` | 更新当前任务状态，如 started、thinking、generating。 |
| `quality_reasoning_chunk` | 追加推理过程文本。 |
| `quality_output_chunk` | 追加优化或评估的输出文本。 |
| `quality_done` | 当前任务完成。 |
| `quality_error` | 当前任务失败。 |

这种设计使质量页不仅能显示最终结论，还能展示“推理过程 -> 输出结果”的动态过程，更适合论文实验和调试场景。

### 提示词与结果缓存

质量页同时维护两类本地缓存：

| 缓存 | 作用 |
|---|---|
| 提示词缓存 | 缓存模板评估与优化提示词。 |
| 评估缓存 | 缓存章节优化结果、评估结果、状态和卡片视图。 |

这使得页面在切换路由或刷新后仍能尽量保留上一次评估状态。

## 模板查看页设计

模板查看页对应 `TemplateViewPage`，用于展示和编辑 `llmkg_templates` 中的章节配置。

### 页面职责

模板页不仅是静态查看工具，还承担模板章节级管理职责，包括读取数据库中的模板章节配置、编辑章节写作约束并回写后端。

### 页面结构

页面由两部分构成：

| 区域 | 作用 |
|---|---|
| 顶部说明卡片 | 说明页面用途、展示当前章节数量和刷新按钮。 |
| 章节卡片网格 | 按章节渲染 `TemplateSectionCard`。 |

### 章节卡片内容

每张模板卡片至少包含：

| 字段 | 含义 |
|---|---|
| 章节编号与标题 | 当前模板章节身份。 |
| `level` 标签 | 章节层级。 |
| `source_type` | 章节内容来源类型。 |
| `kg_field` | 关联的图谱字段。 |
| `fixed_text` | 固定文本内容。 |
| `gen_instruction` | 章节生成要求。 |

管理员用户可进入编辑模式，对 `source_type`、`fixed_text` 和 `gen_instruction` 分别修改，并支持保存或恢复默认值。普通用户则只能查看。

### 模板页布局

模板页支持紧凑布局。开启紧凑布局后，章节卡片使用 `template-card-grid--compact` 样式，以每行 8 个卡片的方式展示，更适合在论文演示或大屏场景下快速浏览整体模板结构。

## 对比实验页设计

对比实验页对应 `ExperimentPage`，是当前前端中最复杂的页面之一。该页面并不只是展示实验记录，而是将实验题库、实验配置、运行进度、结果预览和评估结果组织为统一的实验工作台。

### 页面目标

实验页用于集中验证系统在以下维度上的表现：

1. 输入边界拦截。
2. 故障主体识别与消歧。
3. 图谱与模板约束效果。
4. 多故障链式生成效果。

### 页面顶层结构

页面首先显示实验总览卡片，其中展示：

| 信息 | 来源 |
|---|---|
| 单故障题数量 | 预置题集。 |
| 多故障题数量 | 预置题集。 |
| 数据库实验题数量 | 实验题库接口。 |
| 复用现有 FastGPT 插件标记 | 固定说明。 |

其后按实验计划渲染一组实验卡片。每张实验卡片又支持四个视图：

| 视图 | 作用 |
|---|---|
| `实验信息` | 展示实验目的、输入、预期输出和实验过程图。 |
| `实验控制` | 配置实验次数、并发数，启动生成和评估。 |
| `产出预览` | 查看某次实验运行的输出内容。 |
| `评估结果` | 查看人工或自动评估的结果。 |

### 实验流程图展示

实验页通过 `ExperimentFlowDiagram` 把不同实验组的插件链路、支撑层数据、运行时变化变量和阶段状态可视化为文字流程图。这使实验设计不仅停留在表格说明，而能直观看到对照组与实验组在运行链路上的差异。

### 实验控制台

`ExperimentControlPanel` 提供实验执行控制，包括：

| 控件 | 作用 |
|---|---|
| 实验次数 | 控制每轮运行次数。 |
| 并发数 | 控制实验组并发执行数量。 |
| 评估并发数 | 控制实验评分并发数。 |
| 已保存实验结果选择器 | 选择历史 `runId`。 |
| 刷新记录 | 拉取最新实验运行列表。 |
| 载入结果 | 加载某次实验结果。 |
| 新建生成 | 启动新的实验运行。 |
| 继续生成 | 基于已有 `runId` 继续执行。 |
| 启动评估 | 对已载入实验结果执行评估。 |

页面同时展示实时进度条、当前活动组标签以及各实验组的阶段状态，是一个面向实验观察的控制台界面。

### 实验页数据持久化

实验页不只依赖浏览器内存状态，还会通过 `/api/experiment/page-cache` 将页面状态快照保存到服务端，包括：

| 状态 | 用途 |
|---|---|
| 卡片视图选择 | 记住每个实验卡片当前停留在哪个视图。 |
| 控制台状态 | 保存运行次数、并发数和阶段进度。 |
| 输出状态 | 保存已运行的实验结果。 |
| 评估状态 | 保存当前评估信息。 |
| 选中的运行记录 | 记住用户最近查看的 `runId`。 |
| 评估页紧凑模式 | 记住实验评估展示方式。 |

页面还会从 `/api/evaluation/question-suite` 加载数据库实验题库，从 `/api/experiment/runs` 和 `/api/experiment/run` 加载历史运行记录与运行详情。

因此，对比实验页本质上是一个面向研究验证场景的前端工作台，而不是简单的结果列表。

## 页面间数据复用关系

前端各页面之间并不是孤立存在，而是通过“最近一次生成结果”和“页面缓存”形成数据复用。

| 来源页面 | 去向页面 | 复用数据 |
|---|---|---|
| 预案生成页 | 图谱溯源页 | `question`、`faultScene`、`graphMaterial` |
| 预案生成页 | 质量评估页 | `chapters`、`faultScene`、`graphMaterial` |
| 模板查看页 | 质量评估页 | 提示词与模板结构参考 |
| 实验页 | 评估结果页签 | 实验运行输出、评分与题库信息 |

这种设计使前端表现层围绕“一次生成结果的后续分析与复用”形成闭环，而不是把每个页面都做成独立入口。

## 前端与服务接入层的通信特点

前端当前主要采用三类通信模式：

| 模式 | 适用场景 |
|---|---|
| 普通 JSON 请求 | 登录、模板读取、题库读取、运行记录查询、图谱子图查询。 |
| SSE 流式请求 | 预案生成、质量优化与评估、实验运行。 |
| 本地或服务端缓存 | 最近一次生成结果、页面状态快照、模板缓存、评估缓存。 |

其中，SSE 是前端表现层的关键能力。它使页面能够展示“正在获取基本信息”“正在并行生成章节”“某一章正在追加正文”“案例搜索已完成”等细粒度运行状态。如果改为传统同步请求，页面只能在整篇生成结束后一次性展示结果，无法体现运行时编排层的阶段化特征。

## 论文可用表述

前端表现层采用 React 与 TypeScript 构建，围绕预案生成、图谱溯源、质量评估、模板查看和对比实验组织多页面交互。系统首页并不是直接展示最终结果，而是先通过登录机制建立用户会话，再在应用壳层中统一管理导航、显示偏好、夜间模式和模型余额信息。以预案生成页为核心，前端通过 SSE 调用服务接入层的流水线接口，将后端返回的基本信息获取、模板切片、章节并行生成和案例检索等事件映射为可视化阶段状态，并以章节卡片形式实时展示各章的流式输出。最近一次成功生成的预案结果会持久化到本地，并进一步被图谱溯源页和质量评估页复用，从而形成“生成 - 溯源 - 评估”的连续分析链路。与此同时，模板查看页支持按章节管理模板约束，对比实验页则将实验题库、实验流程、运行控制、结果预览和评估记录整合为统一工作台。通过上述设计，前端表现层将后端复杂的插件编排与知识调用过程转化为连续、可观察、可追踪和可复用的用户交互流程。
