# 第五章案例验证与结果分析写作规划

本文档用于支撑论文第五章“案例验证与结果分析”部分写作。内容仅依据当前项目代码、实验脚本和已落盘实验结果整理，后续若补跑实验，可在对应小节追加统计表和典型输出。

## 1. 实验章节定位

第五章前半部分已承接第四章系统设计，重点说明数据与知识层、运行时编排层、服务接入层和前端表现层的具体实现。实验部分应进一步回答“系统实现是否有效”的问题，因此不宜只写页面功能或截图，而应围绕应急预案生成链路的关键能力展开验证。

建议将实验部分命名为“5.6 案例验证与结果分析”，并按以下逻辑组织：

1. 实验目的与评价维度：说明实验验证对象、对照组和评价标准。
2. 实验环境与测试样例：说明实验页面、变体脚本、样题来源和执行方式。
3. 输入边界判定验证：作为辅助实验，验证系统对非电力、非支持设备或跨设备问题的拦截能力。
4. 设备识别与故障匹配验证：作为核心实验之一，验证主体消歧和二级故障匹配对图谱检索正确性的影响。
5. 图谱增强与模板约束对照验证：作为核心实验之一，验证知识图谱与章节模板对生成内容的贡献。
6. 多故障链式生成验证：作为核心实验之一，验证多故障识别、逐故障图谱检索和融合生成能力。
7. 实验结果小结：围绕上述三个核心实验归纳系统能力验证结果，避免展开局限性和后续改进，相关内容放入第六章。

## 2. 已确认实验能力

实验页面后端位于 `docker/frontend-proxy/services/experiment_service.py`。其中 `PLAN_GROUP_SCRIPTS` 已实现四类实验计划，每类均包含完整工作流对照组和两个消融实验组。实验运行目录默认为 `/app/data/frontend_experiment_runs`，最大并发数为 `15`。

| 实验计划 | 对照组 | 实验组一 | 实验组二 | 验证目标 |
| --- | --- | --- | --- | --- |
| boundary | 主流水线 | 移除输入边界校验 | 关键词边界校验 | 比较 LLM 边界判定、无边界和关键词规则的差异 |
| disambiguation | 主流水线 | 移除主体判定 | 关键词主体判定 | 比较主体消歧策略对设备表和故障节点检索的影响 |
| graphTemplate | 主流水线 | 移除图谱 | 移除模板 | 比较图谱素材与章节模板对内容质量的影响 |
| multiFault | 多故障主流水线 | 单故障普通链路 | 识别多故障但仅检索主故障图谱 | 比较逐故障图谱检索和多故障融合生成的效果 |

变体脚本位于 `scripts/experiment_page_variants/`。`README.md` 已明确脚本对应关系，`common.py` 中实现了实际差异：

| 脚本 | 关键实现差异 |
| --- | --- |
| `boundary_no_boundary.py` | 调用基本信息插件时忽略边界失败结果，继续进入模板切片和章节生成 |
| `boundary_keyword_boundary.py` | 使用设备关键词和故障关键词进行简单边界判断，命中失败时直接输出错误 |
| `disambiguation_drop_subject.py` | 保留故障抽取，但将设备主体置为未明确，并基于弱主体策略重新查询图谱 |
| `disambiguation_keyword_subject.py` | 使用关键词匹配设备主体，再基于候选二级故障选择故障节点 |
| `graph_template_no_graph.py` | 将图谱检索素材置空，保留模板约束和生成步骤 |
| `graph_template_no_template.py` | 将正式章节模板替换为自由组织指令，保留图谱素材 |
| `multi_fault_single_fault.py` | 对多故障问题使用单故障普通链路处理 |
| `multi_fault_no_per_fault_graph.py` | 使用多故障识别结果，但只检索主故障图谱，不提供伴随故障逐项图谱素材 |

## 3. 已有实验材料可提炼结论

### 3.1 轻量实验：裸 LLM 与模板 LLM

材料来源：`docs/对照试验题库/轻量实验结果/20260425_164200/comparison_results.md` 与同目录 JSON。

已落盘样题为：`某110kV断路器同时出现控制回路短路故障和合分闸控制回路故障，请生成一份现场应急处置方案。`

可写入论文的事实：

1. 裸 LLM 输出字符数为 `2024`，带模板 LLM 输出字符数为 `3530`。
2. 两组输出都包含编号步骤和处置动作。
3. 带模板 LLM 按模板生成了“信息报告、先期处置、现场研判、抢修恢复、响应终止”等章节结构，结构规范性明显高于裸 LLM。
4. 裸 LLM 也能给出通用应急流程，但章节组织更偏自由发挥，和系统预设预案结构不完全一致。
5. 带模板 LLM 存在局部编号错误，例如“3.3 现场研判”下出现“3.1.1 原因研判”，可作为质量检查模块必要性的说明依据；不要在第五章扩展为系统局限性讨论。

建议论文表述：模板约束能够显著提升章节完整性和结构一致性，但仍不能完全替代格式审查模块；因此系统后续设置质量评估与格式检查接口具有必要性。

### 3.2 图谱增强对照：有图谱与无图谱

材料来源：`docs/对照试验题库/第四章有无图谱对照/20260424_231700/chapter4_kg_vs_no_kg.md` 与同目录 JSON。

该实验复用“基本信息获取 -> 模板切片 -> 并行生成”链路，只比较第四章“处置措施”。两组差异仅在并行生成阶段：有图谱组传入真实“图谱检索方案素材”，无图谱组传入“知识图谱无数据”。

可写入论文的事实：

1. Q001 断路器拒动场景中，有图谱组标签统计为 `KG=23, GEN=15, FIX=0`，无图谱组为 `KG=0, GEN=34, FIX=0`。
2. Q002 电压互感器熔断器熔断场景中，有图谱组标签统计为 `KG=14, GEN=15, FIX=1`，无图谱组为 `KG=0, GEN=40, FIX=0`。
3. 有图谱组在故障原因、故障现象、应对措施、安全风险和应急资源方面显式吸收图谱素材，例如断路器拒动场景中覆盖直流电源、控制回路、线圈、铁心、机构卡阻、辅助开关、储能异常等原因要点。
4. 无图谱组仍能生成可读内容，但主要依赖模型常识扩写，来源标签全部为 `GEN`，内容可追溯性弱于有图谱组。
5. 有图谱组通过 `[KG]`、`[GEN]`、`[FIX]` 标签保留内容来源，便于后续人工审查和质量评估。

建议论文表述：图谱增强的主要价值不是简单增加文本长度，而是提高故障原因、措施、风险和资源条目的领域相关性与可追溯性。

## 4. 建议评价维度

第五章评价维度应对应第一章既定分析框架，不另起一套无法闭环的新指标。

| 评价维度 | 说明 | 可观察证据 |
| --- | --- | --- |
| 结构规范性 | 输出是否遵循预案模板章节、小节和顺序 | 标题覆盖率、章节顺序、是否出现编号错误 |
| 知识相关性 | 内容是否围绕正确设备、故障和图谱字段展开 | 设备识别结果、故障二级节点、图谱条目命中情况 |
| 措施完整性 | 是否覆盖报告、隔离、研判、抢修、恢复和终止等关键动作 | 章节内容覆盖、应急资源和风险措施是否出现 |
| 内容可追溯性 | 是否能区分图谱、模型生成和固定模板来源 | `[KG]`、`[GEN]`、`[FIX]` 标签统计和图谱素材引用 |

## 5. 建议章节文本框架

### 5.6.1 实验目的与环境

写作重点：说明实验目的不是评价通用大模型能力，而是验证本文系统在电力设备应急预案生成任务中的边界控制、知识增强、模板约束和多故障融合能力。实验环境包括 FastGPT 插件、NebulaGraph 图谱、Python 并行流水线、服务代理和实验页面。

建议字数：500-700 字。

建议图表：实验计划与对照组设置表。

### 5.6.2 测试样例与评价方法

写作重点：说明样题来源包括轻量实验样题、对照试验题库 Q001-Q010 和实验页面题库；评价方法采用结构规范性、知识相关性、措施完整性、内容可追溯性四个维度，同时结合来源标签统计和典型案例分析。

建议字数：500-700 字。

建议图表：测试样例表、评价维度表。

### 5.6.3 输入边界判定实验

写作重点：比较完整工作流、移除边界校验和关键词边界校验。完整工作流依赖 FastGPT 基本信息插件中的边界判定，移除边界校验会让不合规问题继续进入生成链路，关键词规则对显式设备词有效但对语义类、跨设备类问题不稳定。

建议字数：400-600 字。

当前状态：已有代码能力，尚需补跑结果或从既有真实验证记录中补充案例。

### 5.6.4 设备识别与故障匹配实验

写作重点：本节是核心实验之一。说明设备主体消歧直接影响 Nebula space 选择和二级故障候选集合，进而影响图谱检索素材和后续章节生成。应重点比较完整工作流、移除主体判定、关键词主体判定三组结果，突出完整工作流在复杂位置描述和主体歧义场景中的优势。可结合已验证案例说明“主变旁断路器拒动”应识别为断路器，“开关柜内电流互感器二次开路”应识别为互感器，“主变轻瓦斯”识别为变压器，“电缆沟进水”识别为电缆。

建议字数：900-1200 字。

当前状态：已有真实验证结论，但最好补充实验页面批量结果截图或运行记录。

### 5.6.5 图谱增强与模板约束实验

写作重点：本节是核心实验之一。合并轻量实验和第四章有无图谱对照结果。先说明模板对结构规范性的贡献，再说明图谱对知识相关性和可追溯性的贡献。应重点引用 Q001 和 Q002 的标签统计，并把 `[KG]`、`[GEN]`、`[FIX]` 来源标签作为内容可追溯性的主要证据。

建议字数：1000-1400 字。

建议图表：裸 LLM/模板 LLM 对照表、有图谱/无图谱标签统计表、典型输出片段表。

### 5.6.6 多故障链式生成实验

写作重点：本节是核心实验之一，不能省略。比较完整多故障链路、单故障链路和“仅主故障图谱”链路。完整链路使用多故障基本信息插件识别多个故障节点，并对每个故障分别检索图谱素材；单故障链路容易只围绕主故障展开；仅主故障图谱组可以识别多故障但缺少伴随故障的图谱支撑。论文中应重点说明多故障模式下“单故障边界判定 + 多故障图谱聚合”的实现结果：先用单故障插件复用边界控制，再用多故障插件和二级故障查询插件生成逐故障图谱材料，最后由并行生成节点融合为章节内容。

建议字数：900-1200 字。

当前状态：已有代码能力和若干真实接口验证记录，但需补跑实验页面结果以形成表格。

### 5.6.7 实验结果小结

写作重点：只归纳第五章实验已经验证的系统能力，重点围绕设备识别与故障匹配、图谱增强与模板约束、多故障链式生成三个核心实验展开。边界判定作为辅助验证简要总结即可。不要在本节展开样题规模、模型输出错误、复杂语句识别不足等局限性，也不要写后续改进路线；这些内容统一放入第六章。

建议字数：600-800 字。
