# 第二章 相关理论与方法概述

## 2.1 知识图谱相关理论与方法

### 2.1.1 知识图谱建模与构建相关理论

知识图谱是一种以图结构形式组织领域知识的表达方式，其核心思想是将现实世界中的概念、实体及其关系抽象为节点和边，并通过属性对节点和边进行补充描述。相较于传统关系数据库主要依赖二维表结构表达数据对象，知识图谱更擅长刻画复杂对象之间的多跳关联关系，因而更适合描述电气设备故障场景中“故障原因—故障现象—应对措施—故障后果—安全风险—应急资源”这类链式知识结构。

从知识组织范围来看，知识图谱通常可分为通用知识图谱和领域知识图谱两类。通用知识图谱面向开放领域，强调知识覆盖面和常识性表达，常服务于搜索、问答和推荐等场景；领域知识图谱则面向特定行业，强调知识的专业性、准确性和可解释性，更适合支撑电力设备运检、故障诊断和应急处置等高约束任务。本文所构建的电气设备应急知识图谱属于典型的领域知识图谱，其数据来源主要包括设备故障层级梳理文本、典型案例文档、模板章节配置及相关项目中间产物。

结合当前项目实际实现，本文知识图谱采用“根节点—一级故障—二级故障—下游明细节点”的层级结构。图谱中的核心节点类型包括 `root_node`、`fault_l1`、`fault_l2`、`fault_cause`、`fault_symptom`、`response_measure`、`fault_consequence`、`safety_risk` 和 `emergency_resource`。其中，`root_node` 用于表示设备主题入口，`fault_l1` 和 `fault_l2` 分别表示一级故障大类和二级故障小类，而其余明细节点则负责表达与预案生成密切相关的具体知识片段。与之对应的关系类型包括 `contains`、`caused_by`、`has_symptom`、`handled_by`、`results_in`、`has_risk` 和 `needs_resource` 等，用于组织故障链与处置链。

在具体建模过程中，模式层设计不仅要考虑知识表达的完整性，还要兼顾后续任务调用的便利性。当前项目在图谱节点中保留了 `lvl`、`source_id`、`node_desc`、`stroke`、`degree` 和 `weight` 等字段。其中，`lvl` 用于区分知识层级，`source_id` 便于在图谱中进行稳定排序和映射，`node_desc` 则承载可被后续生成直接消费的自然语言描述，而 `stroke`、`degree` 和 `weight` 主要服务于图谱可视化和重点节点突出显示。由此可见，本文知识图谱并不仅是知识存储结构，也是一种服务于检索增强生成与结果溯源的任务型知识组织框架。

当前项目中，不同设备的知识图谱分别围绕高压断路器、输电线路、变压器、互感器、电力电缆、避雷器、环网柜、光缆、杆塔等设备方向展开，并通过统一的实体关系模式保持跨设备知识组织的一致性。这种设计使系统既能够支持单设备故障场景，也为多设备、多故障的复合应急场景提供了知识基础。

### 2.1.2 领域知识组织与知识抽取相关方法

领域知识图谱的构建不仅依赖模式设计，还依赖对原始知识资产的系统整理与结构化抽取。对于电力应急场景而言，知识并非集中存储在单一数据库中，而是广泛分布于设备原理资料、故障层级梳理文本、典型案例、历史预案、知识库导入文件以及中间图谱 Excel 成品之中。这些知识具有明显的多源异构特征，如果不经过预处理直接用于图谱构建和大模型调用，容易出现术语不统一、结构混乱和知识边界不清等问题。

结合当前项目现状，知识组织与抽取过程主要分为三个阶段。第一阶段是原始资料整理与预处理。项目通过对 `txt/`、`txt_rag_cases/`、`xls/成品/`、`知识库导入数据/` 等目录中的资料进行分类整理、文本清洗、结构规范化和格式统一，使不同来源的数据具备可进一步解析的基础。第二阶段是面向图谱建模的层级拆分与字段映射。系统围绕设备、一级故障、二级故障及其下游明细层，使用多套构建脚本将原始文本转化为节点与关系表、中间清单和图谱成品。第三阶段是面向生成任务的知识再组织。图谱中沉淀的实体与关系并不会直接原样输入模型，而是被进一步重组为可供生成调用的知识素材。

当前项目并未采用严格意义上的端到端神经知识抽取模型作为主实现路径，而更多采用“结构化文本梳理 + 脚本构建 + 图谱导入”的工程化方法。这种方法的优点在于：一方面，它能充分利用电力设备知识原本就具有层级化、列表化和规则化的特点；另一方面，它便于研究者对知识组织方式进行人工审校和持续修订，符合当前课题阶段对专业准确性的要求。

需要指出的是，尽管本文没有将 BERT、BiLSTM-CRF 或 Biaffine 等信息抽取模型作为系统核心实现，但“知识抽取”这一概念在本项目中依然是必要的，只是其主要体现在对故障层级文本、案例文本和模板字段的结构化拆分与归一化映射上。从任务本质看，这仍然属于面向电力应急领域的领域知识抽取与知识重组过程。

### 2.1.3 知识图谱检索与推理相关方法

知识图谱检索与推理是知识图谱应用的核心环节，其目标是在结构化知识空间中快速定位与问题相关的节点和关系，并在已有知识基础上形成可供任务使用的结果。在经典研究中，知识图谱推理通常包括基于规则的推理、基于路径的推理、基于语义相似度的推理以及与检索增强生成相结合的推理等方式。

基于规则的推理通常依赖人工编写的 IF-THEN 规则，在图谱中出现符合前提条件的实体和关系时触发结论推导。该方法优点在于推理逻辑清晰、可解释性强，但缺点也非常明显：规则设计与维护成本高，覆盖范围有限，且对图谱数据质量要求较高。结合当前项目看，系统并未构建独立规则库，因此不属于纯规则推理范式。

基于路径的推理主要利用图谱中实体之间的多跳路径关系挖掘隐含知识。例如，通过“故障—原因—现象”或“故障—措施—风险—资源”路径，可以组织更丰富的上下游知识。这种方法与本文项目中围绕二级故障节点向下游节点展开的查询方式有一定相似性，但当前项目主要采取的是定向结构化检索，而不是复杂多跳路径搜索与路径打分，因此也不属于严格意义上的路径推理系统。

基于语义相似度的推理通常通过向量表示或语义嵌入计算实体、关系或问题之间的相似性，进而完成相关知识发现。这类方法在处理模糊匹配和语义扩展时具有优势，但当前项目并未将图谱推理建立在知识嵌入或相似度传播机制之上，因此不构成主要实现方式。

从当前项目实际实现看，最贴切的归类方式是“知识图谱检索增强”。系统首先根据用户问题识别设备对象和二级故障节点，再围绕对应二级故障在 Nebula 图数据库中查询其关联的故障原因、故障现象、应对措施、故障后果、安全风险和应急资源，最后将这些结构化结果组织为生成素材。也就是说，本文项目的图谱使用方式并不是为了在图内推导出大量新知识，而是为了从结构化知识网络中**有方向地检索与当前任务最相关的知识片段**，并将其显式传递给大语言模型。

因此，本文项目在知识图谱推理相关方法上的实际立场可以概括为：以知识图谱检索增强为主，以结构化关系链组织为辅，不采用纯规则推理，也不依赖复杂路径搜索或知识嵌入推理。这种选择与项目目标高度一致，即服务于应急预案生成任务，而不是构建一个面向开放问答的通用图推理系统。

## 2.2 大语言模型相关理论与方法

### 2.2.1 Transformer 与生成式大语言模型

Transformer 模型是当前大语言模型的基础架构，其核心机制在于通过自注意力机制建立序列内部各位置之间的依赖关系，从而突破传统循环神经网络在长距离依赖建模上的局限。Transformer 的提出，使模型能够在更大规模语料上以更高的并行效率进行训练，并为后续大规模参数化语言模型的发展奠定基础。

生成式大语言模型是在 Transformer 架构基础上发展起来的自回归语言模型，其通过对上下文进行条件概率建模，逐步预测下一个最可能的词元，从而生成自然语言内容。当前主流生成式大语言模型具备较强的指令理解、结构化输出、长文本组织和多任务泛化能力，使其能够在复杂业务场景中承担问答、总结、抽取、规划与文本生成等多种角色。

结合当前项目实现，系统中的大语言模型主要承担三个方面的任务：一是进行场景要素提取，例如识别故障二级节点、故障对象和场景约束；二是根据模板和知识素材逐章生成预案内容；三是对生成结果进行格式优化和质量评估。由此可见，模型在本项目中不是单一生成器，而是作为多个工作流阶段中的核心智能组件参与任务执行。

### 2.2.2 提示工程与结构化生成方法

提示工程是当前大语言模型应用中的关键控制手段，其核心思想是通过合理设计系统提示词、任务说明、输入变量和输出约束，引导模型在给定边界内稳定完成指定任务。对于强专业、强结构约束的电力应急预案生成任务而言，提示工程的重要性尤为突出，因为模型不仅需要“会写”，还必须“按正确结构写、围绕正确知识写、按照正确角色写”。

在当前项目中，提示工程主要体现在三个层面。第一层是场景解析提示，负责将自然语言问题转化为故障节点和场景字段。第二层是章节级生成提示，要求模型严格按照模板文本中的章节和小节组织内容，同时根据图谱检索素材和场景要素进行补全。第三层是质量评估与优化提示，负责对原始生成结果进行格式规范化和质量反馈。这样的提示体系已经不再是单一提示词，而是一套与工作流阶段绑定的提示集合。

结构化生成方法则是提示工程在长文本任务中的进一步延伸。当前项目通过模板切片将整份预案拆为多个章节任务，并把“模板章节文本 + 图谱检索素材 + 场景结果”共同作为生成输入，使模型在章节级边界内完成结构化写作。这种方法有效降低了单轮长文本生成的复杂度，也提高了结果的一致性和可控性。

### 2.2.3 检索增强生成相关方法

检索增强生成（RAG）是一种通过外部知识检索提升大语言模型生成质量的常见方法。其基本思想是在生成前先从知识库、文档库或图数据库中召回与问题相关的知识片段，再将这些知识作为上下文提供给模型，从而缓解模型幻觉，提高内容的事实一致性和专业针对性。

与传统文档型 RAG 相比，当前项目采用的是“结构化图谱知识 + 章节模板约束 + 大语言模型生成”的复合式检索增强机制。一方面，系统从 Nebula 图数据库中检索围绕二级故障节点组织的结构化知识；另一方面，前端和流水线还支持案例检索能力，用于补充非结构化经验材料。二者共同构成了生成阶段的知识支撑来源。

因此，本文项目中的检索增强生成并不是简单的“文档召回 + 模型回答”，而是一种围绕特定章节生成任务所组织的任务型 RAG。其重点不只是召回知识，而是让召回结果在模板边界和场景约束下转化为可直接生成预案的章节素材。

## 2.3 知识图谱与大语言模型结合相关理论与方法

### 2.3.1 知识图谱增强大语言模型生成

知识图谱增强大语言模型生成的核心思想，是将图谱中的结构化知识作为显式外部知识源注入模型，从而提高模型输出在事实一致性、术语准确性和内容可解释性方面的表现。相较于仅依赖模型参数记忆的纯生成方式，图谱增强生成能够显著降低领域任务中的知识漂移和幻觉风险。

结合当前项目实现，这种增强主要表现为：系统先围绕设备和故障节点查询知识图谱，再将查询结果组织为 `图谱检索方案素材`，随后与场景解析结果、模板文本一起送入并行生成插件。也就是说，图谱在这里并不是附属参考，而是章节生成时的核心事实支撑来源之一。

从实验结果看，这种图谱增强不仅提高了输出文本中设备级、部件级和故障链级细节的密度，也增强了结果的可溯源性。前端还通过 `KG / GEN / FIX` 等来源标记和图谱溯源页面，让用户能够看到知识片段如何参与最终生成。

### 2.3.2 模板驱动与工作流编排方法

模板驱动方法强调在生成前为文本提供明确的章节框架、字段边界和内容类型约束。对于应急预案任务而言，模板不仅是格式骨架，更是将固定内容、知识填充内容和模型补全内容进行区分的重要机制。当前项目中的模板字段包括 `source_type`、`kg_field`、`fixed_text` 和 `gen_instruction` 等，它们共同构成章节级模板配置。

工作流编排方法则强调把复杂任务拆分为多个可管理、可追踪的阶段执行。当前项目的主流水线围绕“基本信息提取—模板切分—并行章节生成—案例检索—质量评估”展开，前端通过 SSE 事件流持续接收阶段状态和章节增量文本。工作流编排使系统能够把知识检索、章节生成和结果评估组织为一个连续、可追踪的执行过程，从而显著提高系统稳定性和可调试性。

因此，在本文项目中，模板驱动和工作流编排并不是附加功能，而是实现知识增强长文本生成的结构控制机制。没有模板，生成结果容易漂移；没有工作流，复杂任务难以稳定执行。

### 2.3.3 多故障场景下的知识聚合与生成方法

在电力应急场景中，事故往往不是单一故障触发，而是多个故障链条交织形成的复合事件。因此，单故障检索和单轮生成方法很难覆盖多故障语义融合的要求。针对这一问题，当前项目在场景解析阶段引入 `主故障二级节点` 与 `故障二级节点` 数组，并在后端通过逐故障并行检索与聚合机制形成多故障图谱素材。

这种方法的核心并不在于简单“把多个故障拼在一起”，而在于在检索和生成层同时保持主次关系：主故障负责维持预案主线，其他故障作为伴随故障、诱发因素、叠加风险或补充措施参与章节生成。通过这种聚合机制，系统能够在多故障场景下兼顾结构稳定性和内容完整性。

因此，多故障场景下的知识聚合与生成方法，是本文项目区别于单故障预案生成系统的重要特征之一，也是知识图谱与大语言模型结合在复杂应急场景中的实际延伸。

## 2.4 原型系统相关支撑技术

### 2.4.1 图数据库与查询网关技术

当前项目采用 Nebula Graph 作为图数据库运行环境，并通过 HTTP 网关将 nGQL 查询封装为可被工作流节点直接调用的服务接口。图数据库负责承载结构化故障知识，HTTP 网关则负责将图查询能力服务化，使前端代理、流水线脚本和 FastGPT 工作流都能够以统一方式调用图谱知识。

这种“图数据库 + 查询网关”的组合，使知识图谱不再停留于静态存储层，而成为能够实时参与任务执行的在线知识服务。

### 2.4.2 前端交互与图谱可视化技术

前端原型系统基于 React、Vite、Ant Design 和 G6 实现，围绕预案生成、图谱溯源、质量评估和模板查看构建完整交互链路。预案生成页支持流式章节展示，图谱溯源页支持故障节点和上下游关系高亮展示，质量评估页支持章节级优化和评估对照，模板页支持模板字段查看和管理。

这种前端能力使系统不仅能输出结果，也能展示知识来源与生成过程，增强了方法的可解释性和演示性。

### 2.4.3 质量评估与结果优化技术

质量评估与结果优化是当前项目的重要后处理能力。系统在预案生成完成后，可按章节执行格式优化、原文评估和优化后评估，通过专门提示词和流式输出完成对结果质量的二次分析。该机制能够在不改变基础知识支撑链路的前提下，提高文本表达的规范性和一致性。

从方法论上看，这一能力使预案生成从“一次生成”转变为“生成—评估—优化”的闭环过程，增强了系统在研究和工程应用中的实用价值。
