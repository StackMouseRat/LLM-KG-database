# 现有工作流生成链路梳理

## 1. 入口

当前前端实际入口为：

- `frontend/src/App.tsx`
- `frontend/server/proxy.mjs`

前端通过 `/api/pipeline/run` 调用：

- `scripts/run_parallel_generation_pipeline.py`

该脚本当前确认的主生成链路是三段式：

1. 基本信息获取
2. 模板切片
3. 并行生成

说明：

- `frontend` 页面里还预留了 `case_search`、`trace`、`quality` 等后续能力。
- 但当前 `run_parallel_generation_pipeline.py` 实际落地的主生成脚本只明确执行上述三段。

## 2. 阶段一：基本信息获取

插件信息：

- 应用名：`基本信息获取`
- 类型：`plugin`
- `appId`：`69e868ff614a51c203e6eca9`
- 发布版本：`2026-04-24 07:19:29`

主输入：

- 插件输入键：`用户问题`

核心处理：

1. 设备识别
2. 故障类型分析
3. 图谱检索与清洗

主输出：

- `用户问题`
- `故障类型分析`
- `图谱检索`
- `知识库名`

在主流水线脚本中的标准化字段：

- `用户问题`
- `故障与场景提取结果`
- `图谱检索方案素材`
- `模板文本`

## 3. 阶段二：模板切片

插件信息：

- 应用名：`模板切片`
- 类型：`plugin`
- `appId`：`69e8755e614a51c203e6feb9`
- 发布版本：`2026-04-22 16:07:06`

这一阶段不是 LLM 生成，而是：

1. 通过 HTTP 节点访问 `llmkg_templates`
2. 读取模板章节配置
3. 通过 Python 代码节点组装章节文本

主输出：

- `切分结果`

在主流水线脚本中的标准化结构：

- `template_id`
- `template_name`
- `current_version`
- `chapter_count`
- `chapters[]`

每章字段：

- `chapter_no`
- `title`
- `section_count`
- `template_text`

## 4. 阶段三：并行生成

插件信息：

- 应用名：`并行生成`
- 类型：`plugin`
- `appId`：`69e86795614a51c203e6e609`
- 发布版本：`2026-04-24 15:05:59`

插件输入键：

- `用户问题`
- `故障与场景提取结果`
- `图谱检索方案素材`
- `模板`

插件输出键：

- `template`

在 `scripts/run_parallel_generation_pipeline.py` 中，每章都会构造如下变量后调用该插件：

```json
{
  "用户问题": "...",
  "故障与场景提取结果": "...",
  "图谱检索方案素材": "...",
  "模板": "当前章节模板文本"
}
```

主流水线用线程池逐章并行调用该插件，并通过流式事件回传：

- `chapter_started`
- `chapter_chunk`
- `chapter_done`

## 5. 当前生成链路的关键特征

1. 不是“用户问题直接进最终生成”。
2. 先经过设备识别和故障场景结构化。
3. 再通过模板系统把长文本拆成章节级任务。
4. 最终生成插件拿到的不是裸问题，而是“问题 + 场景提取 + 图谱素材 + 当前章节模板”。

这意味着对照实验至少可以拆出以下版本：

- 裸 LLM：只给原问题
- 模板 + LLM：给原问题和模板
- KG + 模板 + LLM：给原问题、场景提取、图谱素材和模板
- 当前完整工作流：三段串联 + 并行章节生成
